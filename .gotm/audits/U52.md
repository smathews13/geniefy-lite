### MEDIUM severity
- **Progressive-persistence test is weaker than its own claim** (`tests/test_api.py:121` `test_run_persists_progressively`): the body comment says "start_run save + tracer span-exit save + final save ‚Üí > 1" but the assertion is `store.save_calls >= 2`. The actual count is **3** (verified by running the path); start_run(1) + final save(1) alone already satisfy `>= 2`. So if the `PersistingTracer`'s per-phase save were silently broken ‚Äî e.g., the tracer never wired into the orchestrator, or its `_save` reduced to a no-op ‚Äî `save_calls` would drop to 2 and the test would still PASS. D18's "progressive per-phase persistence" is therefore under-guarded: a regression that removes the *per-phase* save (the thing the "watch it think" U9 poll depends on) would not be caught by this suite. A broken poll, a missing concurrency guard, and a review no-op ARE all caught (see Meaningfulness); only this one D18 property is asserted too loosely.
  - Suggested fix unit: add a test that fakes a multi-phase orchestrator and asserts `save_calls >= phases + 2` (or asserts the tracer's save fires once per `span`), so a dropped per-phase save fails. Frozen-output ‚Üí new follow-on unit, not an edit to U52.

### LOW severity
- **REST surface collapses create+run into one endpoint vs. the LLD's two-step** (`api.py:197` `POST /api/run`). LLD-app-backend ¬ß6 specifies `POST /sessions` (create, returns `status=created`) then `POST /sessions/{id}/run` (start, 202, flips to `profiling`). The target merges these into a single `POST /api/run` (create + enqueue bg task in one call). Defensible v1 simplification ‚Äî a created-but-unrun session isn't needed ‚Äî and all async/poll/202 semantics are preserved; noting only as a deviation from the cited surface.
- **`POST /api/run` returns `status: "created"` rather than D18's "flips to `profiling`"** (`api.py:203`). The bg task may not have started when the response returns, so reporting `created` is arguably more honest than claiming `profiling`; the poll endpoint surfaces the real phase as the worker advances. Cosmetic vs. the literal D18 wording.
- **`/api/` path prefix** on every route vs. the LLD's bare `/sessions/...`. Reasonable namespacing, not a fidelity break; the verbs/resources/status-codes all match.
- **Ledger Recent-updates says "18 hermetic TestClient tests"** (`.gotm/LEDGER.md:129`) but the file defines **14** `test_*` functions (all pass). The count is wrong in the living Recent-updates log, not in the U52 unit row; harmless but worth correcting if the log is relied on. (The "240 passed" full-suite figure in the same line is now 253 after U53 ‚Äî expected drift, not a U52 error.)
