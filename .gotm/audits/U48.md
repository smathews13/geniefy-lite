## Test-meaningfulness assessment

- **Dropped jsonb field** ‚Äî the round-trip test (`test_save_then_load_round_trip`) compares
  `loaded.to_dict() == original.to_dict()`, which is symmetric: a field dropped on *both*
  `to_dict` and `from_dict` would slip through. But that drift lives in `state.py` (audited
  under U27/U28 with an enum-parity guard); for *store.py*'s own behavior the symmetric compare
  is adequate because store does not transform fields ‚Äî it persists `to_dict` and rehydrates via
  `from_dict` verbatim. A field dropped *by the store* (e.g., if it serialized a subset) WOULD be
  caught, because the snapshot captured at param[8] is asserted to contain `target` and
  `table_draft.proposed_comment` and to fully reconstruct. Adequate for this unit. (LOW-1.)
- **Missing delete** ‚Äî `test_save_issues_upsert_rewrite_and_commit` asserts both DELETEs are
  present AND that `column_drafts` is deleted before `table_drafts` (FK ordering). A removed or
  reordered delete would fail. Caught.
- **Non-guarded schema** ‚Äî `test_rejects_unsafe_schema` asserts `StoreError` on
  `"geniefy; drop table"`; `test_rejects_bad_target` asserts `StoreError` on an unqualified
  target. Caught.
- D38 uuid-or-NULL is exercised indirectly (the fixture uses `template_id="default"`, a
  non-uuid) but **no test directly asserts `params[6] is None`** for the non-uuid case nor that a
  real uuid passes through unchanged ‚Äî see MED-2.

## Findings

### HIGH severity
- none.

### MEDIUM severity
- **MED-1 ‚Äî schema guard is good but qualified identifiers are never *validated*, only
  parameterized.** `_split_target` (store.py:46-50) strips backticks and splits on `.`; the three
  parts are bound as params, so injection is impossible, but a part containing a stray `.` or
  empty segment is only loosely checked (`not all(parts)` rejects empties; a 3-part requirement is
  enforced). This is safe for SQL injection (bound params) and matches U2's "three columns" model,
  so it is not a correctness bug ‚Äî flagged only because the module docstring/Check language implies
  identifier *guarding*, while only the **schema** is regex-guarded and the target parts rely on
  parameterization. No fix strictly required; documenting the distinction would prevent a future
  reader from assuming the target parts are identifier-validated.
- **MED-2 ‚Äî D38 (the unit's entire reason for existing) has no direct unit test.** No test in
  `tests/test_store.py` asserts that `_uuid_or_none("default")` ‚Üí `None` is written to
  `sessions.template_id` (param[6]), nor that a genuine uuid string is preserved. The behavior is
  exercised only transitively (the fixture happens to use a non-uuid name and the round-trip
  passes because the name lives in the jsonb). A regression that wrote the raw name to param[6]
  would NOT be caught by the hermetic suite ‚Äî exactly the gap the U47 live test had to catch the
  first time. Suggested fix unit: add two assertions ‚Äî `params[6] is None` for the
  `template_id="default"` fixture, and `params[6] == "<uuid>"` for a uuid-valued `template_id`.

### LOW severity
- **LOW-1 ‚Äî symmetric round-trip compare can mask a both-sides field drop.** `to_dict()==to_dict()`
  in `test_save_then_load_round_trip` would not catch a field dropped symmetrically in
  `state.py`'s serde. This is a `state.py` concern (already tracked as U29 test-hardening for U27),
  not a store.py defect; noting it here only because the store's round-trip test inherits the same
  shape. No new action beyond U29.
- **LOW-2 ‚Äî test comment param-layout is slightly mislabeled.** `tests/test_store.py:119`
  comment lists the params but the prose label for index 5 ("status") vs the code at line 123
  (`params[5] == "reasoning"`, the session_status rollup) is fine; the layout comment is accurate.
  No action. (On re-read this is correct ‚Äî withdrawn; left as a note that the param indices were
  re-verified and match.)
