## Findings

### HIGH severity
- **`_await_terminal` is NOT wired into `databricks_sql_execute` (the SP read runner) ‚Äî only into `make_obo_sql_execute`.** `providers.py:237` is `return _rows_from_statement_response(resp)` with no `_await_terminal`, whereas `providers.py:266` is `return _rows_from_statement_response(_await_terminal(w, resp))`. The unit claim, the U145 ledger row, and the recovery log (LEDGER.md:14/186/246) all state the fix was wired into **both** runners. It is not. Worse, the SP runner is the one that feeds the `Profiler` (the "large-table profile" the spec explicitly names as the slow-query case), the `ContextGatherer` (lineage/query-history), and table enumeration (api.py:583 / 593 / 599 / 624‚Äì627); the OBO runner only handles the UC write-apply path (api.py:622). So the exact silent-`[]` masking U145 set out to fix ‚Äî a slow profile/enumeration statement still PENDING/RUNNING past the 50s `wait_timeout` returning `[]` ‚Äî remains live on the primary path, while the polling fix sits on the path least likely to hit it. Confirmed: a RUNNING response passed to `_rows_from_statement_response` returns `[]` (masking). The unit cannot be trusted as done.
  - Suggested fix unit: change `databricks_sql_execute`'s `execute` to `return _rows_from_statement_response(_await_terminal(w, resp))` (mirroring make_obo_sql_execute:266), and add a hermetic test asserting the SP runner polls a RUNNING response to terminal (parallel to the OBO behavior).

### UNVERIFIED
- **`max_polls=150 √ó poll_seconds=2s` ‚âà 5-minute ceiling.** Reasonable as a stuck-statement guard, but unjustified against any documented warehouse statement-timeout expectation; no source ties the 5-min cap to a profiling SLA. Could be too short for a genuinely large-table profile, or too long as a hang guard. Flagged for the practitioner ‚Äî not a defect in scope.
- **Malformed-response forward-progress.** If `get_statement` repeatedly returns a non-terminal status (state stuck at RUNNING with no progress), `_await_terminal` does not loop forever ‚Äî it is the capped path and raises after `max_polls` (verified by `test_await_terminal_gives_up_on_stuck_statement`). No infinite-loop risk. `statement_id` is read via `getattr(resp, "statement_id", None)` (lines 190/195), which matches the SDK `StatementResponse.statement_id` attribute (confirmed present). The one residual edge: if a response is non-terminal but lacks `statement_id` (None), `get_statement(None)` would be called ‚Äî would surface as an SDK error, not a silent hang; acceptable but unasserted.

## Summary
HIGH: 1 ¬∑ MEDIUM: 0 ¬∑ LOW: 0 ¬∑ UNVERIFIED: 2
