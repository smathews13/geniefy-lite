## U95 — regenerate grays only that card

### Findings
None above the trivial bar.

Verified against the two flagged failure modes:

1. **No all-card graying on single-card regen.** `ReviewView.doRegen(targets)` adds only the
   passed targets to the set (`setRegenerating(prev => new Set([...prev, ...targets]))`,
   ReviewView.tsx:51). Per-card button calls `doRegen([c.target])` (single element, line 106);
   DraftCard grays only when `regenerating={regenerating.has(c.target)}` (line 107). So exactly
   one card grays. "Regenerate all" calls `doRegen(cards.map(c => c.target))` (line 69) → all
   targets → all gray. The `disabled={anyRegenerating}` on the all-button (line 70) only disables
   that button; it does not gray cards. Correct per-card vs all-card behavior.
2. **No flip to RunView during regen.** `App.tsx:30` derives `hasDrafts`; RunView renders only
   `IN_FLIGHT && !hasDrafts` (line 124), ReviewView renders `DONE || (IN_FLIGHT && hasDrafts)`
   (line 130). During a regenerate drafts already exist → ReviewView stays mounted, RunView is
   skipped. Correct.

`doRegen` branch (ReviewView.tsx:52): `targets.length >= cards.length ? {all:true} : {targets}`
— `all:true` only when every card is targeted; single-card always takes the `{targets}` path.
Correct. The set-clear `useEffect` (lines 27-30) clears when status transitions out of the
in-flight set; `prevStatus` ref is updated every effect run, so it observes the in-flight→terminal
transition under the 1.5s poll. Live Chrome DevTools verification confirmed graying clears after a
single-column regen with no view flip. No defect.

DraftCard.tsx no longer owns `useRegenerate` (it takes `onRegenerate`/`regenerating` props,
lines 41-42, 50-51); `useRegenerate` is now called once in ReviewView (line 19). The gray styling
(`pointer-events-none animate-pulse opacity-60`, line 72) + "Regenerating…" pill (lines 83-87)
gate on the `regenerating` prop. Matches spec.

**Verdict: PASS**
