## Findings

### HIGH severity
None. The keystone control flow is correct:

- **Phase order (U4 §4).** `run` sequences `PROFILING → GATHERING → REASONING → JUDGING → GATING` (orchestrator.py:83-102; REASONING/JUDGING phase set inside `reasoner.draft`/`judge.score`, GATING in `_gate`). The phase cursor is written on the state through the loop, terminating at `AWAITING_INPUT` (interactive + questions) or `READY_FOR_REVIEW`. Mutation "skip gathering" was **caught** by the suite.
- **Resume does NOT re-run expensive phases (D17).** `resume` (orchestrator.py:105-112) only merges answers + `_redraft` the affected drafts; profiling/gathering are not touched. Probe: profile-provider call count and context-provider call count are **identical** before and after `resume` (1 → 1 each). Mutation "resume calls `run()`" was **caught**.
- **No draft corruption on merge.** `_redraft` (orchestrator.py:152-173) builds a sub-state with only the affected columns, re-drafts/-judges them, and merges back via `state.column_drafts = [fresh.get(c.column_name, c) for c in state.column_drafts]` — un-affected drafts keep their original object identity and content (probe P2 confirmed `o_orderkey`/`o_total` object-id stable and comment unchanged; only `x` re-drafted). The merge is correct.
- **Failure path fabricates nothing (U4 §8).** A `ProfileError` (`permission_denied`/`timeout`/…) returns `Failed` with `phase=FAILED`, `table_draft is None`, `column_drafts == []`, and **gather is never called** (orchestrator.py:84-88, `_fail` 126-128; probe P3). Mutation "fabricate a table draft on failure" was **caught**.
- **Interactive vs batch not swapped.** `_gate` (orchestrator.py:115-124) returns `NeedsInput` only when `mode == INTERACTIVE and questions`; batch always reaches `ReadyForReview` and the Gate marks tripped drafts `low_confidence` (probe P4/P5). Mutation "swap interactive/batch" was **caught** (4 tests red).
- **Hermetic (D1).** Imports are stdlib only (`contextlib`, `typing`) + sibling `geniefy_core` modules (`gate`, `profiler`, `state`). No third-party, no I/O, no network/SQL. All model/IO is behind the injected components. **Confirmed hermetic.**
- **Decision fidelity.** D8 — the Gate routes on `draft.confidence`, which the Judge overwrites with its authoritative weighted `overall` (judge.py:107); the orchestrator does not re-derive confidence. D17 — `SessionState` stays serializable throughout (no live objects stashed on it). D2 — interactive single-table loop intact.
- **Cross-reference integrity.** Component method calls match the real APIs: `profiler.profile(target, sample=SampleSpec(...))`, `gatherer.gather_into(state)`, `reasoner.draft(state)`, `judge.score(state)`, `gate.apply(state, signals)`. `_schema_meta_dict` emits `{primary_key, foreign_keys:[{columns,referenced_table,referenced_columns}], partition_columns, columns:[…]}` — matches what `_signals` reads (`foreign_keys[].columns`) and what Judge/Reasoner consume (`state.schema_meta or {}`). The MLflow span names `profile`/`gather_context`/`reason`/`judge`/`gate` match U4 §6; the `NullTracer` no-op default is structurally present with the real tracer deferred to the app layer (as the prompt expected — not a finding).

### MEDIUM severity
- **U42 does not assert un-affected drafts survive `resume` (test-meaningfulness gap):** `tests/test_orchestrator.py:150-163` (`test_resume_answers_then_ready`) asserts on `x` (re-drafted) and that the `user_answer` snippet landed, but never asserts that `o_orderkey`/`o_total` still exist after resume. Mutation #5 — replacing the careful merge with `state.column_drafts = list(sub.column_drafts)` (which would drop every un-affected draft, leaving only `x`) — **passed all 7 tests (NOT caught)**. The orchestrator's own logic is correct (proven by probe P2), so this is a gap in the *tests*, not a bug in U41. Because the audit prompt explicitly named "resume corrupting un-affected drafts" as a defect class the integration tests should catch, this is flagged MEDIUM rather than LOW.
  - Suggested follow-on unit: add an assertion to `test_resume_answers_then_ready` (or a new test) that after `resume`, `res2.state.column_draft("o_orderkey")` and `"o_total"` are still present and unchanged.

### LOW severity
- **Unused imports.** `DraftKind` and `DraftStatus` are imported in `orchestrator.py:29-30` but never referenced (the orchestrator only sets `Phase`; status/kind assignment lives in the Gate). Cosmetic; remove on the next polish pass.
- **`reason`/`judge` spans are coarse vs U4 §6.** U4 §6 specifies `reason` "one per batch" and `judge` "one per draft"; the orchestrator wraps the whole `reasoner.draft`/`judge.score` calls in a single span each (orchestrator.py:97-100). The per-batch/per-draft granularity lives inside those components but is not separately spanned at this layer. Structurally the spans exist (which is all the prompt asked to confirm); the finer granularity is a deferred observability detail.
- **`enum_no_decode` Gate rule not exercised by the integration suite.** `_signals` reads `col.get("is_enum_candidate")` (orchestrator.py:200) — a real, contract-defined U3 profile field (LLD-profiling-tool.md:113), so the read is correct. But `FakeProfileProvider` never sets it, so one of the Gate's three force-low rules never fires in U42. The rule is covered by the Gate's own unit suite (per the ledger); this is only an integration-coverage gap. Optional: add a profile column with `is_enum_candidate: true` and no decode source to one integration test.
