## Findings

### Check 1 — append_audit is correct  ✓
- INSERT targets `{s}.audit_log` (store.py:139) where `s = self._schema`, qualified at
  `__init__` via `_qualify` (store.py:58), which raises `StoreError` on non-identifier
  schema. No raw/interpolated user input in the SQL — only the validated schema name is
  formatted in; all data goes through `%s` placeholders. No injection.
- Column list `(session_id, draft_kind, draft_id, action, actor, before, after)`
  (store.py:140) matches the migration's insertable `audit_log` columns and order exactly
  (001_init.sql:201–211, excluding the `id`/`created_at` defaults).
- `before`/`after` cast to jsonb via `%s::jsonb` (store.py:141) with `_jsonb(before)` /
  `_jsonb(after)` (store.py:143); `_jsonb` returns `None` for `None` else `json.dumps`.
- Own transaction: `append_audit` calls `self._conn.commit()` (store.py:145), independent
  of `save`'s commit — append-only, as the docstring and D21 require.

### Check 2 — Applier wiring is right  ✓
- `append_audit` is called exactly once, at apply.py:124–127, inside the APPLIED branch
  *after* `_write` succeeds (apply.py:113–122). Every non-write outcome returns earlier:
  conflict (apply.py:93–95), empty-over-empty noop (100–101), empty-over-existing refused
  (102–104), idempotent noop (107–110), write failure (114–116). So the call is
  unreachable on conflict / skipped_noop / failed / empty-refused. Verified against the
  full control flow.
- `action="applied"` — a valid `audit_action` enum member (001_init.sql:59–61).
- `draft_kind="table" if col is None else "column"` — both valid `draft_kind` enum members
  (001_init.sql:55). Table draft (col is None) → "table"; columns → "column".
- `actor=created_by` — the apply caller threaded through `apply(...)` → `_apply_one`.
- `before={"comment": live}` / `after={"comment": proposed}` — live (re-read) vs proposed
  comment, matching D21's before/after change snapshots.

### Check 3 — tests bite  ✓
- `PYTHONPATH=src /tmp/geniefy_pgvenv/bin/python -m pytest tests/test_store.py
  tests/test_apply.py -q` → **29 passed**.
- `test_append_audit_inserts_row` asserts one `INSERT INTO geniefy.audit_log` + commit,
  with params[0]=session_id, params[1]=draft_kind, params[3]=action, params[4]=actor,
  params[5]=before jsonb, params[6]=after jsonb — the right columns/values.
- `test_apply_appends_audit_row_per_written_item` asserts 3 applied audit rows (table + 2
  cols), kinds {"table","column"}, actor on all, and before=None/after="x" on a column.
- `test_no_audit_row_for_noop_or_conflict` asserts a noop + conflict append nothing.
- Mutation probes (applied to /tmp copies, restored):
  1. Drop the `append_audit` call → `test_apply_appends_audit_row_per_written_item`
     FAILS. ✓ bites.
  2. Move the call into the idempotent-noop branch → `test_no_audit_row_for_noop_or_conflict`
     FAILS. ✓ bites.
  3. Swap the *params-tuple* before/after values (`_jsonb(after), _jsonb(before)`) →
     `test_append_audit_inserts_row` FAILS. ✓ bites on value order.
  4. (informational) Swapping only the SQL *column-name comment text* in the column list
     does NOT fail any test — see LOW finding below.
