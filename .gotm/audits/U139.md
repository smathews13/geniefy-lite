### UNVERIFIED
- **Non-terminal states (PENDING/RUNNING) are not treated as errors.** The installed SDK's `StatementState` has six members: `CANCELED, CLOSED, FAILED, PENDING, RUNNING, SUCCEEDED`. The fix raises only on the three failure-terminal states; `PENDING`/`RUNNING` fall through and return `[]` (verified: `state="RUNNING"` → `[]`). Both callers use `wait_timeout` (default `"50s"`, providers.py:199/216) and do **not** poll for completion, so a statement that has not finished within the wait window would return a non-terminal status and silently yield `[]` — the same empty-result masking class as the original U139 bug, just for slow rather than failed statements. This is a latent edge, not the bug U139 targeted (the U138 reproduction was a synchronous PARSE error → FAILED), and the schema-run queries are lightweight `information_schema` reads unlikely to exceed 50s. **Severity is LOW-to-UNVERIFIED**: it does not block U139's claimed scope (raise-on-failed-terminal-state) and per the audit instruction I am flagging, not fixing. Recommend a follow-on unit if long-running statements are ever expected on this path (treat any non-SUCCEEDED state — including PENDING/RUNNING after wait_timeout — as an error, or poll to a terminal state).
