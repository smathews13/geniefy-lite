## Findings

### HIGH severity
- none.

The key correctness point holds: the **`overall` is computed by the Judge itself** as the
template-rubric-weighted sum (`judge.py:105` — `overall = round(sum(subscores[d] * weights[d] for d in dims), 4)`),
reading weights from the U25 `Template.rubric.weights()` — it is **not** trusted from the model
(the model is asked only for `subscores` + `issues`, `judge.py:82`). `draft.confidence` is then
overridden with that `overall` (`judge.py:107`), the authoritative-confidence path per D8. A
missing/invalid dimension defaults to **0.0** (penalized, not rewarded — `_clean_subscores`,
`judge.py:137-140`), subscores are clamped to [0,1] (`judge.py:140`), ERROR drafts are skipped
(`judge.py:57,62-63`), and a failed judge call is isolated to its own draft (`judge.py:97-101`).
Hermetic: all model access is via the injected `LLMClient` (D1).

### MEDIUM severity
- **`test_overall_is_template_weighted_not_mean` does not actually distinguish a weighted
  sum from a plain mean** (`tests/test_judge.py:87-91`). Its input is `(completeness=1,
  specificity=1, grounding=0, template_conformance=1)`, which evaluates to 0.75 under *both*
  the weighted sum (.30+.30+0+.15) **and** the arithmetic mean ((1+1+0+1)/4). I verified by
  mutation: replacing the weighted sum with a plain mean leaves this test **green**. The
  weighted-vs-mean distinction is in fact caught — but only incidentally, by
  `test_subscores_clamped_and_missing_defaulted` (its 0.425 case is asymmetric, so mean=0.375
  ≠ weighted=0.425). The net code coverage is sound, but the test that *advertises* the
  weighted-overall guarantee is mislabelled and would not fail under the very regression its
  name targets. This is a U38 (test) finding, not a U37 code defect.
  - Suggested fix unit: strengthen `test_overall_is_template_weighted_not_mean` with an
    asymmetric case whose weighted sum differs from its mean (e.g. completeness=1, others=0 →
    weighted 0.30 vs mean 0.25), so the test name's guarantee is actually enforced.

### LOW severity
- **`_fit` (D32 evidence-trim) is best-effort, not a hard budget enforcer**
  (`judge.py:110-124`). It trims the `context` snippets to a placeholder count when the prompt
  exceeds `max_input_tokens`, then returns the rebuilt messages **without re-checking** that the
  trimmed prompt is now under budget, and `complete_json` is called **without** passing
  `max_input_tokens` (`judge.py:96`), so the `LLMClient`'s `TokenBudgetExceeded` guard never
  fires for the Judge. If the load-bearing draft + direct profile entry alone still exceed the
  budget, the over-budget call is sent anyway. This is consistent with D32's "surfaced honestly,
  load-bearing parts kept" intent and the design note that the full compaction pipeline lives in
  the Reasoner/Budgeter (U24), so it is acceptable for the Judge — but it is a single-shot
  reducer, not the bounded guarantee the word "budget" implies. `test_large_evidence_trimmed_
  under_budget_but_still_scores` (`tests/test_judge.py:165-170`) only asserts a score is still
  produced; it does **not** assert the resulting prompt is under the limit (it cannot, given the
  above), so the test name slightly over-claims.
  - Suggested fix unit (optional): document the Judge's single-reducer behavior in the
    docstring, or (when U24's Budgeter lands) route the Judge prompt through it for a true bound.
- **`HeuristicTokenCounter` import in the test is unused** (`tests/test_judge.py:17`) — it is
  passed to `LLMClient` in `_judge`, so this is fine; noting only that the budget test relies on
  the heuristic 4-chars/token estimate, which is intentional for hermeticity.

### UNVERIFIED
- **§6 "one judge span per draft" (MLflow tracing).** The design calls for one `judge` span per
  draft. `judge.py` makes one model call per non-error draft (`_score_one` per draft), which is
  the structural precondition, but the module emits **no MLflow span** itself — span emission is
  presumably the Orchestrator's job (U4 §6 lists spans at the orchestration layer, and U37's
  scope is the scorer, not tracing). Could not verify the span is actually opened anywhere
  because the Orchestrator unit is not yet built. Flagging so the Orchestrator audit confirms a
  `judge` span wraps each `_score_one` call.
