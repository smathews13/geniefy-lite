## Findings

### HIGH severity
- None.

The gap is closed. `build_service` (`api.py:291-294`) now constructs the `ContextGatherer`
from `build_context_providers(sql, config.mcp_providers, databricks_mcp_session)` — it no
longer ignores `config.mcp_providers`. The factory `databricks_mcp_session` is imported and
passed through. Confirmed by static read + the regression simulation above.

### MEDIUM severity
- **`token_budget` field in the D34/U24 §3 schema is not surfaced or consumed.** The D34
  registration schema and LLD U24 §3 list a per-provider `token_budget` (feeds the §2
  compaction); neither the documented `app.yaml` example (`app/app.yaml:38-50`) nor
  `McpContextProvider` (`context.py:181-213`, which honors only `max_snippets`) uses it.
  Snippets are budget-trimmed globally by `ContextGatherer._trim` (`context.py:286-295`), so
  there is no correctness hole, but the documented per-provider `token_budget` knob is inert.
  This is a pre-existing `McpContextProvider` (U39) limitation surfaced by U63's example, not
  a regression introduced here. Non-blocking.
  - Suggested follow-on unit: honor per-provider `token_budget` in `McpContextProvider`, or
    record a decision that global `context_token_budget` subsumes it and drop the field from
    the U24 §3 schema / app.yaml docs.

### LOW severity
- **`build_service` itself is not directly unit-tested.** No test imports `build_service`
  (`tests/` references it only in comments). The unit's own docstring + D36 + D39 state the
  real `build_service` wiring is integration-verified at deploy (lazy infra imports), and the
  load-bearing delegate it calls — `build_context_providers` — IS unit-tested and the
  `config.mcp_providers → build_context_providers` call is statically confirmed. So the
  app.yaml→AppConfig→gatherer→invoke path is proven end-to-end via `AppConfig.from_env` +
  `build_context_providers` (`test_app_yaml_config_flows_to_providers`,
  `test_gatherer_uses_configured_mcp_provider`); only the final glue line inside
  `build_service` rests on integration verification. Acceptable per the unit's sanctioned
  deploy-time boundary; noted for completeness.
- **`app.yaml` example omits the optional `token_budget` key** present in the U24 §3 example.
  Cosmetic doc-vs-spec drift; tied to the MEDIUM above.

### UNVERIFIED
- **Concrete `databricks_mcp_session` connect against a live MCP server.** Requires the `mcp`
  client lib (absent here) + a live Databricks managed-MCP endpoint. The unit explicitly
  defers this to deploy-time integration verification ("don't fail on" per the audit brief and
  D39) — flagged, not a finding.
- **Runtime config UI / provider-CRUD endpoints.** Explicitly deferred by D39 for v1; not in
  scope for this unit.

## Spec / oracle conformance (checklist)
1. **Gap closed (CRITICAL):** YES. `build_service` wires `config.mcp_providers` into the
   `ContextGatherer` via `build_context_providers` (`api.py:291-294`). Regression-caught.
2. **`build_context_providers`:** built-ins (`uc_lineage`, `query_history`) always present and
   first; one `McpContextProvider` per **enabled** entry; `enabled=False` skipped (default
   `True`); hermetic — imports only `geniefy_core.context`, no SDK. Verified by
   `test_builtins_plus_enabled_mcp` + `test_no_mcp_when_none_configured`.
3. **`mcp.py` factory:** returns `_DatabricksMcpSession` with `list_tools()` + `call_tool()`
   (satisfies the `McpSession` protocol, `context.py:172-178`). Auth = secret-scope token
   (base64-decoded secret value), else in-platform OAuth token (`mcp.py:35-48`) — no raw
   secret in config/logs (D4/D5). Lazy-imports `databricks.sdk` and the `mcp` client inside
   methods; module imports SDK-/mcp-free (verified: `import-ok`).
4. **app.yaml:** surfaces `GENIEFY_MODEL_ENDPOINT` (with sonnet/opus comment) + a documented
   `GENIEFY_MCP_PROVIDERS` JSON example matching D34; valid YAML; secret-scope auth only.
5. **Tests meaningful:** 4 tests prove app.yaml→`AppConfig.from_env`→`build_context_providers`
   →`ContextGatherer.gather`→**invokes** the configured server (snippet returned, no warnings),
   and that disabled providers are skipped. Both regressions (ignoring `mcp_providers`; wiring a
   disabled provider) would be caught.
