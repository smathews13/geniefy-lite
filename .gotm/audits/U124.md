## What was checked

### 1. LIKE pattern correctness — the literal `__` match (HIGH-risk item)
PASS. The default branch (`api.py:266-267`) appends, via a Python **raw** string:
`AND table_name NOT LIKE '\_\_%' ESCAPE '\'`.
Verified the raw string resolves to exactly that SQL (the bytes are `\_\_%` with `ESCAPE '\'`). At the SQL layer, `ESCAPE '\'` makes the two `\_` sequences match **literal underscores**, and the trailing `%` matches any suffix — so the predicate excludes names that begin with a literal double-underscore (`__materialization_mat_*`, `__…`). It does **not** degrade to the buggy form where an unescaped `_` is a single-char wildcard (which would over-match any 2-char prefix). The HIGH failure mode named in the spec ("not escaping `_` so it matches any 2-char prefix") is **not** present.

### 2. SQL-injection regression — none
PASS. The new clause is a **fixed literal** with no caller value interpolated — `include_internal` is a boolean gate only; its value never enters the SQL. Catalog still goes through `_q_ident` (`api.py:333-335`), schema and `name_like` still through `_q_lit` (`api.py:338-340`). Confirmed at runtime: a `name_like` of `o'brien_%` is emitted as `LIKE 'o''brien_%'` (properly doubled), unchanged from the U110 posture. No injection surface added.

### 3. Clause composition with skip_documented + name_like + max_tables
PASS. Generated a combined query at runtime; the clause order is valid SQL:
`… WHERE table_schema = 'sch' AND table_type IN ('MANAGED','EXTERNAL') AND table_name NOT LIKE '\_\_%' ESCAPE '\' AND table_name LIKE 'fact_%' ORDER BY table_name`.
The internal-exclusion clause sits after the base WHERE and before the optional `name_like` clause, both ahead of `ORDER BY`. `skip_documented` (Python-side filter, `api.py:273-281`) and `max_tables` (Python slice, `api.py:282-284`) are unaffected and compose correctly. With `include_internal: true` the clause is omitted and the rest of the query is intact.

### 4. Test meaningfulness
PASS. `test_enumerate_tables_excludes_internal_by_default` (`test_api.py:503-511`) captures the emitted SQL via `_fake_enum_runner` and asserts (a) default call → SQL contains both `NOT LIKE` and `ESCAPE`; (b) `include_internal: true` → SQL contains **neither** `NOT LIKE`. This is a real present/absent assertion on the generated SQL string for both branches, not a trivial smoke test. The pre-existing `test_enumerate_tables_skips_documented_and_applies_filters` still asserts catalog/schema/pattern escaping, so the U110 injection posture remains covered.
