## Findings

### HIGH severity
- None. The suite runs green (31/31), and the round-trip hardening provably catches the
  symmetric-drop class that motivated the U27 MED. Nothing blocks downstream consumption.

### MEDIUM severity
- **The `from_dict` omitting-dict default path for draft statuses is still untested —
  the second sub-point of the U27 MED is not fully closed.** U27.md's MED had two parts:
  (a) the symmetric round-trip blind spot, and (b) "No test exercises the `from_dict`
  defaults for `apply_status` / `status` from an omitting dict" — its suggested fix
  explicitly called for `ColumnDraft.from_dict({"column_name": "c"})` /
  `TableDraft.from_dict({})`. U29's `test_table_draft_defaults` (:177) exercises the
  **dataclass constructor** default (`TableDraft(...)`), not the
  `_draft_common_from_dict` on-disk default path. Mutation-confirmed gap: changing
  `_draft_common_from_dict`'s `apply_status` default from `"not_applied"` to `"applied"`
  (state.py:255 — the path a pre-`apply_status` or partial persisted blob takes) leaves
  **all 31 tests green.** Part (a) is closed; part (b) survives. Non-blocking — the
  module's `from_dict` default is correct today; this is a residual regression-net gap.
  - Suggested fix unit: add `TableDraft.from_dict({})` / `ColumnDraft.from_dict({"column_name":"c"})`
    asserting `status == DRAFT` and `apply_status == NOT_APPLIED` (distinct from the
    constructor-default path U29 now covers).

### LOW severity
- **`test_draft_kind_has_table_and_column` pins member NAMES, not enum values against the
  migration.** It asserts `{k.name ...} == {"TABLE","COLUMN"}` but does not add
  `(DraftKind, "draft_kind")` to `test_enums_match_migration`'s parametrization
  (tests/test_state.py:47-54), which is the actual schema-drift guard. So a *value* drift
  (e.g. `TABLE = "tbl"`) with the member name intact would pass this test while breaking
  serialization into `geniefy.draft_kind`. The U27 LOW finding's literal suggested fix
  (add the parametrize tuple) was not taken; U29 chose a name-set assertion instead. The
  member set is pinned, which is a real improvement; the value↔schema parity remains
  unguarded. (Verified manually the values match today: `'table','column'`.)
  - Suggested fix: add `(DraftKind, "draft_kind")` to the `test_enums_match_migration` parametrize list.

### UNVERIFIED
- None. All claims were checkable by running the suite and mutating the source.
