## Findings

### HIGH severity
- None.

  The two dispatch-flagged HIGH risks were checked and **cleared**:
  - **Stack vs D31.** §1's table is an exact, substitution-free match to D31:
    Vite + React + TypeScript · Tailwind CSS + shadcn/ui · TanStack Query · visx (bespoke)
    + Recharts (routine). "No global state lib" matches D31 verbatim. No contradiction.
  - **Serve/build model on Databricks Apps.** §4 line 61 states the SPA is built by
    `vite build` in the deploy orchestrator **before** `bundle sync`, and that "the
    Databricks App runtime does **not** run `npm build`"; FastAPI serves the pre-built
    `app/static` via a StaticFiles mount with an SPA fallback. This is the correct
    Databricks Apps model (the App runtime runs only the `app.yaml` command; it does
    not build the frontend) and is consistent with U24 §4 (`deploy.sh` step 2 = build
    frontend → `app/static`, step 4 = `bundle deploy`). Would work on Apps. Not HIGH.

### MEDIUM severity
- None.

### LOW severity
- **API path-prefix self-inconsistency (`/api/*` vs bare `/sessions/{id}`).**
  §4 (line 60) mounts the API "under `/api/*` (U5)", but §5 (line 67) polls
  `GET /sessions/{id}` and the U5 oracle (`LLD-app-backend.md` §6) specifies bare
  paths (`/sessions`, `/sessions/{id}`, `/templates`, `/providers`) with no `/api`
  prefix. The two are internally inconsistent and §4's `/api/*` is not grounded in
  U5. Harmless in practice (the api client carries whatever prefix is chosen, and the
  static mount is at `/`), but it should be reconciled so the served route map is
  unambiguous.
  - Suggested fix unit: a one-line reconciliation note (frontend build unit) — either
    drop the `/api` prefix to match U5's bare paths, or record the prefix as a
    deliberate frontend-side namespacing of the U5 routes.

- **`paused` session status is unhandled by the poll loop.** §5's poll-loop status
  set is poll-while `{created, profiling, gathering_context, reasoning, applying}` and
  stop-on `{awaiting_input, ready_for_review, applied, failed, cancelled}`. The
  canonical `session_status` enum (U2 §5, line 232-234) additionally contains
  **`paused`** (U5 §6 exposes `POST /pause`). A `paused` session falls into neither
  set, so the poll loop neither continues nor recognizes it as a stable/terminal
  state. Minor — `paused` is "implicit (state already persisted)" per U5 §6 and the
  query layer would simply idle — but the enumerated sets are not exhaustive over the
  enum.
  - Suggested fix unit: frontend build unit — treat `paused` as a stop/idle state in
    the `refetchInterval` predicate (or note it explicitly as out-of-poll).

- **`D2/D8` citation for deferred component tests is loose.** §7 line 82 defers
  component tests "(with the broader test/eval harness, D2/D8)". D2 defers the
  *expanded eval harness* and D8 is the *LLM-judge / MLflow* decision — neither is
  squarely about frontend component tests. The intent (tests are deferred with the
  broader harness) is fine; the specific D-refs are a slightly approximate hook.
  Cosmetic; fix only if convenient.

### UNVERIFIED
- None. All cited `D#`/`U#`/§ references resolved against on-disk oracles. U2's
  `session_status` enum (consulted to validate §5's status set) confirms `applied` and
  `cancelled` are genuine session-level terminal statuses — so §5's stop-set is
  correct, not a per-item/session conflation as it might first appear.
