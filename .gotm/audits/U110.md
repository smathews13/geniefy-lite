### LOW severity
- **No pre-trigger schema-existence validation in `start_schema_run`** (api.py:238–253). LLD §5 prose says `POST /api/schema-runs` "validates the schema exists (SP, information_schema), inserts a `schema_runs` row … triggers the Job." The implementation skips the pre-trigger existence check and goes straight to `create_schema_run` → trigger; enumeration (and any implicit existence check) happens inside the Job (§2/§6 diagram + D51 option-b refinement, which moves enumeration into the Job task). This is consistent with the §2/§6 "Job enumerates" model and not a behavioral defect — a non-existent schema simply yields an empty/failed run — but it diverges from the literal §5 sentence. Optional: a cheap `enumerate_tables` pre-check at the route to fail fast (R4 scope-confirm), or align the §5 prose.
- **`start_schema_run` leaves status `enumerating` when no job is triggered** (api.py:245,251). If `run_schema_job is None` (not wired) or returns a falsy `job_run_id`, the record stays `enumerating` with no error returned to the client. Harmless given option-b wiring is always present in `build_service`, and a hermetic test path exercises the `None` case indirectly; only cosmetic.

### UNVERIFIED
- **Live SDK Job-trigger / cancel** (`_make_run_schema_job`, `_make_cancel_schema_job`, api.py:648–681). Explicitly out of scope per the audit brief — these are lazy-SDK, integration-only. Confirmed they are *lazy* (the `from databricks.sdk import WorkspaceClient` import is inside the inner `run`/`cancel` closures, not at module load; module `py_compile` and the full hermetic suite both pass with the SDK absent) and that they do **not** break `build_service` (only constructed, not called). Their real behavior against a live workspace → U113.

## Detailed verification (checks from the brief)

- **`run_hands_off` never applies / never writes UC (api.py:283–295):** the method only `_with_mode(..., HANDS_OFF)` → builds a `SessionState` → `store.save(..., schema_run_id=)` → `make_orchestrator(tracer).run(...)` → `store.save(result.state)` → `return result.kind`. No `self.applier`, no `Applier`, no DDL. Grepped the orchestrator: its only "apply" tokens are `gate.apply()` (routing, not UC) and a comment confirming "Apply is never done by the orchestrator in ANY mode." VERIFIED — no apply path.
- **HANDS_OFF persists `awaiting_input`, doesn't pause (U108 orchestrator._gate, orchestrator.py:137–160):** `produces_questions = mode in {INTERACTIVE, HANDS_OFF}`; when questions exist it sets `Phase.AWAITING_INPUT` and returns `needs_input` (the *caller* persists + continues; no block). Matches §3 table. `run_hands_off` returning `result.kind` therefore yields `ready_for_review` | `needs_input` | `failed`. VERIFIED.
- **`schema_run_id` linkage persists (api.py:291 + store.py:92–137):** initial `save` carries `schema_run_id`; the tracer's progressive saves (line 292) and the final save (line 294) omit it. U109 `save`'s `INSERT … ON CONFLICT (id) DO UPDATE SET …` does **not** list `schema_run_id`, so the initial value is preserved across updates (column comment + code confirm). VERIFIED.
- **`enumerate_tables` SQL-injection posture (api.py:255–281):** queries `<catalog>.information_schema.tables` via `self.sql_runner` (SP). Catalog → `_q_ident` (backtick-quoted, backticks doubled); schema + `name_like` → `_q_lit` (single-quote-escaped, quotes doubled); `max_tables` → `int()`-cast. Verified empirically: `_q_lit("o'brien") → 'o''brien'`, `_q_lit("x' OR '1'='1") → 'x'' OR ''1''=''1'`, `_q_ident("cat\` ; DROP …") → \`cat\`\` ; DROP …\``. No raw interpolation of catalog/schema/pattern. `skip_documented` (default True) drops rows whose comment is non-empty after `.strip()` (covers the `"  "` blank-comment case in the test). `max_tables` caps. Raises `SessionError(501)` when `sql_runner is None`. VERIFIED — injection prevented.
- **`start_schema_run` (api.py:238–253):** `create_schema_run` → on trigger, records `job_run_id` + advances `status="running"`; on trigger exception → `finalize_schema_run(status="failed")` + raises `SessionError(502)`; `created_by=identity.actor` (human, D48 attribution) passed from the route. VERIFIED; tests `test_start_schema_run_creates_record_and_triggers_job` (asserts trigger called with catalog/schema, status→running, job_run_id=9999, created_by recorded) and `test_start_schema_run_job_failure_marks_failed_502` (asserts 502 + status→failed) assert real behavior.
- **Routes (api.py:440–474):** `POST /api/schema-runs` → 202; `GET /api/schema-runs` → list; `GET /api/schema-runs/{id}` → 404 when absent, attaches sessions; `POST /api/schema-runs/{id}/cancel`. The request body field `schema` is aliased: `schema_name: str = Field(alias="schema")` + `model_config = {"populate_by_name": True}` — so it doesn't shadow `pydantic.BaseModel.schema`. A client sending `"schema"` works: `test_start_schema_run_creates_record_and_triggers_job` posts `{"catalog":..., "schema":"sch", ...}` and the trigger receives `("cat","sch")`. VERIFIED.
- **`get_schema_run` (api.py:300–306):** returns the run + `run["sessions"] = list_sessions(schema_run_id=run_id, limit=200)`. VERIFIED; `test_get_schema_run_returns_run_with_sessions_or_404` asserts the run echo, `counts`, a `sessions` list, and a 404 on absent.
- **`cancel_schema_run` (api.py:308–320):** best-effort `cancel_schema_job(job_run_id)` only if both wired + present (swallows its exception), then `finalize_schema_run(status="cancelled")`; 404 when the run is absent. VERIFIED; `test_cancel_schema_run_marks_cancelled` asserts status→cancelled + the job-cancel was attempted with id 7.
- **`build_service` wiring (api.py:617–623):** `sql_runner=sql` (the SP `WarehouseSqlRunner`), `run_schema_job=_make_run_schema_job(config)`, `cancel_schema_job=_make_cancel_schema_job()`. `_make_run_schema_job` triggers the **existing** `geniefy_setup` job (`job_parameters={"task":"schema_run", "schema_run_id", "catalog", "schema", "filters"}`) — option b, no new bundle resource. SDK import is lazy (inside the closures). VERIFIED lazy + non-breaking (module compiles + full suite passes SDK-absent).
- **Store-signature parity (U109):** every call matches: `create_schema_run(catalog=, schema=, created_by=, filters=)`, `update_schema_run(run_id, status=, job_run_id=)`, `finalize_schema_run(run_id, status=)`, `get_schema_run(run_id)` (returns dict with `job_run_id`), `list_schema_runs(limit=, offset=)`, `list_sessions(schema_run_id=, limit=)`, `save(state, created_by=, schema_run_id=)`. No signature mismatch → no 500 risk. VERIFIED.
- **D48 identity boundary — no new OBO-token path:** the schema-run route uses only `identity.actor` (human email/user/anon for attribution, `created_by`), never `identity.access_token`. `RequestIdentity.actor` returns the human, never the SP string. Enumeration + `run_hands_off` run as the SP (`sql_runner` = SP runner; orchestrator providers are SP). D51 (more recent, scoped to extend D48) explicitly authorizes recording the *human* as `schema_run.created_by` (attribution-only) while the Job body runs as SP — so route-level `RequestIdentity` extraction here is correct and does NOT violate the D48 "identity only at /apply" refinement (that refinement governs session `created_by`; D51 is the authoritative extension for schema-run attribution). VERIFIED — no D48 violation.
