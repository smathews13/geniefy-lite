### MEDIUM severity

- **"Pin `profile_schema_version`" responsibility only half-realized; `PROFILE_SCHEMA_VERSION` constant is dead.** U4 §3.2 and U3 §4.2 say the Profiler should *normalize to the §4.2 schema and pin `profile_schema_version`*. The module **validates** the incoming version (`_check_version`, profiler.py:593–605, via `_min_version`/`MIN_SUPPORTED_PROFILE_VERSION`) but never **pins/stamps** a canonical version onto its normalized output — `TableProfile.from_dict` (profiler.py:266–272) copies `profile_schema_version` through verbatim from whatever the provider sent. The dedicated constant `PROFILE_SCHEMA_VERSION = "1.0"` (profiler.py:39) is defined, exported in `__all__`, and **never referenced anywhere** in the module — a dead symbol that signals the "pin" verb was intended but not wired. Functionally the gate guarantees the passed-through version is compatible, so downstream is not misled; but the stated responsibility ("pin") is not literally implemented and the unused constant is misleading.
  - Suggested fix unit: append a small follow-on — either stamp `merged["profile_schema_version"] = PROFILE_SCHEMA_VERSION` (or assert-equal) in `_merge`/`profile` so the output carries the core's canonical version, or delete the unused constant and adjust the docstring to say "validate" rather than "pin."

- **Non-conformant provider responses surface as raw `KeyError`, not structured `ProfileError` (U3 §9/§10).** `ColumnProfile.from_dict` (profiler.py:197, `name=d["name"]`) and `ColumnMeta` construction in `SchemaMeta.from_dict` (profiler.py:327, `name=c["name"]`) index `name` directly. A provider that returns a column dict without `name` (a §4.2 violation) raises a bare `KeyError` out of `Profiler.profile` instead of a `ProfileError` — confirmed by exercise (traceback at profiler.py:197 via `TableProfile.from_dict`). U3 §10(d)/§9 call for structured errors over "partial/garbage profiles," and the module's own contract is "structured `ProfileError`, never a fabricated profile." The good path is unaffected and the spec mandates conformant providers, so this is a robustness/defense gap rather than a correctness bug — MEDIUM.
  - Suggested fix unit: append a follow-on to wrap `from_dict` parsing (or the `_call_provider` return) so a missing required field maps to `ProfileError("provider_error"/"unsupported", …)`; cover in U18 tests.

### LOW severity

- **`_merge` dedups columns keyed on `name`, collapsing unnamed columns.** `_merge` (profiler.py:579–585) tracks `seen_cols` by `col.get("name")`; if a (non-conformant) provider omits `name`, multiple distinct columns across batches collapse to one (`seen_cols={None}`) — verified: two unnamed cols merge to 1. Same root cause as the MEDIUM above (required field missing); harmless for conformant providers (`name` required by §4.2). Cosmetic/defense.
  - Note: fold into the same robustness fix unit as the MEDIUM above.

- **Two unused `typing` imports.** `Callable` and `Iterable` are imported (profiler.py:34) but never used. Lint-only.

- **Docstring micro-mismatch on batching default range vs constant.** U3 §7 suggests "25–50 columns per call"; U4 §7 fixes `profile_batch_size` default = 50. The module uses 50 (`DEFAULT_PROFILE_BATCH_SIZE`, profiler.py:43) — correct against U4. No action; noted only for completeness.
