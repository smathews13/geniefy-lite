## Finding-closure check (NF1–NF3)

### NF1 (MED) — reconcile `apply_status` with U6's `pending` — **CLOSED**
U10.md New Finding 1 (`U10.md:60-62`): U10's F2 introduced `apply_status` (default `not_applied`) but its "supersedes" scope omitted U6 §6 (`GET /apply-status`, returns `…/pending` — `LLD-review-apply.md:64`) and U6 §8 (error table), leaving two names for the not-yet-applied state.
U11 §NF1 (`LLD-amend-002.md:11-18`) closes this: it explicitly "supersedes U6 §6 + §8 vocabulary, and extends U10 F2"; declares the per-item apply axis is **exactly** the U10 F2 enum `not_applied | applied | conflict | failed | unsupported | skipped_noop`; maps U6 §6's `pending` → `not_applied`; maps U6 §8's error rows → `apply_status='failed'`; and asserts one source of truth (the `apply_status` column).
- **Verified against U6:** U6 §6's endpoint emits `applied/conflict/failed/pending` (`LLD-review-apply.md:64`) — `pending`→`not_applied` is the exact rename the finding asked for. U6 §8's "Warehouse / DDL error → mark `failed`" and "Missing MODIFY → clear error" rows (`LLD-review-apply.md:77-78`) map coherently to `apply_status='failed'`. Concrete and sufficient.

### NF2 (LOW) — which `apply_status` a no-op records — **CLOSED**
U10.md New Finding 2 (`U10.md:64-66`): U6 §8 resolves the idempotent no-op to "mark `applied`" (`LLD-review-apply.md:39,76,80`) while U10 added a distinct `skipped_noop`; which one the *apply axis* records was unstated.
U11 §NF2 (`LLD-amend-002.md:20-24`) closes this: a no-op records **`apply_status='skipped_noop'`** (not `applied`), explicitly supersedes U6 §8, with a sound rationale (`applied` = "we wrote it"; a no-op wrote nothing — matters for audit trail + readiness) and a correct UX reconciliation (for U9 readiness both `applied` and `skipped_noop` count as resolved, so the glow-up still completes).
- **Verified against U6 §8** (`LLD-review-apply.md:76,80`, "Proposed == live → no-op; mark `applied`") — the superseded text is real; the resolution is consistent with U10 F2's note that `draft_status='applied'` is set for `applied` **or** `skipped_noop` (`LLD-amend-001.md:28`). Closes the ambiguity.

### NF3 (LOW) — `RunConfig.profiling_provider` origin — **CLOSED**
U10.md New Finding 3 (`U10.md:68-70`): U10 F4 cited `config.profiling_provider` as "(U4 §7, D17)", implying U4 §7's `RunConfig` already lists it; it does not — the field is being introduced.
U11 §NF3 (`LLD-amend-002.md:26-30`) closes this: states `profiling_provider` is a **new** field added to `RunConfig` by Amendment 001 (F4), not pre-existing; the `RunConfig` field set = U4 §7's list **plus** `profiling_provider`; cites the three provider options (in-app default / external MCP server / UC function) to D25.
- **Verified against U10 F4** (`LLD-amend-001.md:42`, which does introduce `config.profiling_provider`) and **D25** (`DECISIONS.md:351`: in-app profiling by default, pluggable to external MCP server or UC function) — the D25 citation is accurate. Concrete and sufficient.
