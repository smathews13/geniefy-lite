## Detailed verification log

**Check 1 — accuracy → PASS.**

*(a) Leads with `./deploy.sh` as one command.* The intro (`README.md:15`) reads: "Clone it and run **one command** — [`./deploy.sh`](deploy.sh) — to stand up the App and everything it needs in any workspace". The Deploy-steps header (`README.md:29`) reiterates "One idempotent, re-runnable command — preflight → build the frontend → ensure Lakebase → `bundle deploy` → apply migrations → print the grants you need → print the App URL". This 7-step summary matches `deploy.sh`'s actual steps (preflight/build/Lakebase/deploy/migrate/grants/URL, `deploy.sh:11-18` and the seven `step "n/7 …"` blocks). Accurate.

*(b) Flags match the arg parser.* README shows two invocations:
- `./deploy.sh -t dev -p fe-vm-classic` (`README.md:35`)
- `./deploy.sh -t prod --host https://<your-workspace>.cloud.databricks.com -p <profile>` (`README.md:38`)

`deploy.sh`'s arg parser (`deploy.sh:39-49`) accepts exactly `-t`, `-p`, `--host`, `--skip-build`, `--use-job`, `-h|--help`. Every flag the README uses (`-t`, `-p`, `--host`) is accepted, each requires a value (handled by `needval`), and `TARGET` defaults to `dev` while `prod` is a recognized target in `databricks.yml:43-52`. The host value form (`https://<…>.cloud.databricks.com`) is consistent with the `host` var default in `databricks.yml:23`. No flag is used that deploy.sh would reject. Accurate — both commands would run as written (modulo the `<repo>`/`<your-workspace>` placeholders the operator fills in).

*(c) Host via `-p`/`DATABRICKS_HOST`, not `--var host`.* `README.md:41`: "The **workspace host comes from your CLI profile (`-p`) or `DATABRICKS_HOST`** (`deploy.sh --host` exports it) — Databricks auth fields can't take a bundle variable." This is corroborated by all three oracles: `deploy.sh:57-62` ("the workspace host comes from the profile or DATABRICKS_HOST … so --host exports DATABRICKS_HOST rather than passing --var host"; the code does `export DATABRICKS_HOST="$HOST"`); `databricks.yml:18-23` (the `host` variable is "Informational only — the actual deploy host comes from the CLI profile (-p) or DATABRICKS_HOST"); and `databricks.yml:47-50` / `:56-58` (dev/prod both note host/auth come from profile or DATABRICKS_HOST, not the bundle). A repo-wide read of the README's Deploy section shows **`--var host` does not appear anywhere** — the stale form is not resurfaced. Accurate. The README's raw-CLI fallback (`README.md:43`: `databricks bundle deploy -t <target> -p <profile>` then `databricks bundle run geniefy_setup -t <target>`) likewise carries no `--var host`. Accurate and runnable.

*(d) `geniefy_setup` wired + references `run_migrations.py`.* `README.md:43` gives the raw-CLI equivalent as "`databricks bundle deploy -t <target> -p <profile>` then `databricks bundle run geniefy_setup -t <target>`". `README.md:47` (Database setup): "**`deploy.sh` does this for you** (step 5) via [`migrations/run_migrations.py`](migrations/run_migrations.py) … The bundle also declares a **`geniefy_setup`** job (`databricks bundle run geniefy_setup`) that runs the same migration on a job cluster." Cross-check: `databricks.yml:86-104` defines `resources.jobs.geniefy_setup` with a `migrate` task running `spark_python_task: python_file: ./migrations/run_migrations.py` and a `psycopg2-binary` dependency — i.e., the job **exists and is wired**, and it does invoke `run_migrations.py`. `migrations/run_migrations.py:9-11` confirms its dual invocation ("locally by `deploy.sh` step 5 … and on a Databricks job cluster by the `geniefy_setup` job"). `deploy.sh:116-128` (step 5) runs `run_migrations.py` locally by default, or `databricks bundle run geniefy_setup` under `--use-job`. The prior "planned but not yet wired" language (the U22-era state) is **gone** — the README now describes the job as a present capability, matching disk. Accurate; the `databricks bundle run geniefy_setup` command would work as written.
