## Findings

### HIGH severity

### MEDIUM severity

- **`geniefy_setup` job described in present tense but is not in the bundle** (`README.md:43`). The README's Database-setup section says: *"(A `geniefy_setup` bundle job that runs migrations on deploy is wired in the deployment unit.)"* The current `databricks.yml` has **no `jobs:` section and no `geniefy_setup` job** (verified — grep returns nothing; the bundle defines only `resources.apps.geniefy`). The phrase "is wired" is present-tense and the parenthetical's "in the deployment unit" is easy to read as "already wired by the deploy you're about to run." A customer could expect migrations to run automatically on `bundle deploy`. This does **not** rise to HIGH because the README's *operative* instruction is correct and self-contained: the Database-setup paragraph directs the customer to apply `migrations/001_init.sql` manually via `migrations/APPLY.md` (a validated runbook, U17), so a customer following the README will not have a failed/blocked deploy — they get a working DB the documented way. The defect is honesty/clarity vs. the current bundle (the audit brief flags this exact point: the bundle "has no jobs defined" and the README "should note that job is not yet wired"). The intent traces to design (U8 §3/§7 sketch a `geniefy_setup` Job), but the design is illustrative/deferred, not yet built.
  - Suggested fix unit: append a follow-on README edit that marks the job as *planned / not yet wired* (e.g. "A `geniefy_setup` bundle job to run migrations on deploy is **planned** (not yet in `databricks.yml`); until then, apply migrations manually per `migrations/APPLY.md`."). README is a living governance doc, so this is an in-bounds edit rather than a frozen-output violation — but track it as a unit for write-back discipline.

### LOW severity

- **`<repo>` placeholder in the clone command** (`README.md:30`: `git clone <repo> && cd geniefy-v3`). The repo URL is a literal placeholder. Acceptable for a pre-publication repo and consistent with U8 §1's `git clone …` sketch, but a customer copy-pasting the block hits a non-runnable first line. Cosmetic; fix when the public GitHub URL is known.
- **Database name (`geniefy`) / `database` bundle variable not surfaced** (`README.md:39` overridable-vars list). D30 records that the app working store is a dedicated Postgres **database** `geniefy` and notes a `database` bundle variable (default `geniefy`) should be added when the app resource binding is wired. The bundle does not yet have that variable, so the README correctly omits it — but the Prerequisites/DB-setup section doesn't mention the dedicated `geniefy` database convention (D30) that `migrations/APPLY.md` relies on. Non-blocking: `APPLY.md` (linked) carries the full convention. Note only.

### UNVERIFIED
- None. All oracle files were present and readable; all five check areas were fully verifiable on disk.
