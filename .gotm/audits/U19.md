### MEDIUM severity

- **U14's missing-`name` MED is resolved on the provider/profile path but NOT on the introspector path it also explicitly named.** The U14 MED (`.gotm/audits/U14.md`:20) named **two** call sites that index `name` directly: `ColumnProfile.from_dict` (profiler.py:197) *and* `ColumnMeta` construction in `SchemaMeta.from_dict` (profiler.py:328, `name=c["name"]`). U19 fixed the first: a provider response with a `name`-less column dict now raises `ProfileError("provider_error", "profile response contains a column with no 'name'")` from `_merge` (profiler.py:582–587), which runs *before* `TableProfile.from_dict` — verified by exercise (ProfileError, not KeyError). But the introspector path is still unguarded: `_introspect` (profiler.py:526–537) wraps only the `introspect()` *call* (line 530) in its `try/except`; `SchemaMeta.from_dict(...)` is on line **537, outside** the try block. A `SchemaIntrospector` returning a column dict without `name` therefore raises a **raw `KeyError`** out of `Profiler.profile` — confirmed by exercise (`KeyError('name')`). This is the same robustness/defense gap the U14 MED described, against the module's own "structured `ProfileError`, never…" contract (U3 §9/§10). It is MEDIUM, not HIGH: the happy path and conformant providers/introspectors (the spec mandates conformance) are unaffected, introspection is documented "best-effort structural enrichment," and no downstream consumer is misled by a silent bad profile — it is a loud (if unstructured) failure.
  - Suggested fix unit: append a follow-on that either moves `SchemaMeta.from_dict(...)` inside `_introspect`'s `try` block (so a malformed introspector result maps to `ProfileError`), or guards `name=c["name"]` at profiler.py:328 with a structured error. Mirror the `_merge` guard's code/message.
