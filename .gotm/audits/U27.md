## Mutation testing the U28 suite

To judge whether the tests are *meaningful*, I mutated the module and re-ran the suite:

| Mutation | Caught? |
|---|---|
| Break the rollup (drop `JUDGING` from the rollup map) | **Yes** — 2 tests fail |
| Enum drift (add bogus `ApplyStatus.BOGUS`) | **Yes** — parity test fails |
| Drop `mlflow_run_id` from `from_dict` only (asymmetric) | **Yes** — round-trip test fails |
| **Drop `mlflow_run_id` from BOTH `to_dict` and `from_dict` (symmetric)** | **No** — all 29 still pass |
| **Wrong `from_dict` default for `apply_status` (`"applied"` instead of `"not_applied"`)** | **No** — all 29 still pass |

The suite is largely meaningful (it catches rollup breaks, enum drift, and asymmetric field drops), but it has a **structural blind spot**: the round-trip tests assert `obj2.to_dict() == obj.to_dict()` where the right side is itself `obj.to_dict()`. That comparison is symmetric, so a field dropped from *both* serialize and deserialize, or a wrong *deserialization default*, passes undetected. This is a property of the test, not a defect in U27 — the module's serialization is verified correct today (check 6). The risk is purely that a *future* regression to the round-trip could slip past U28.

## Findings

### HIGH severity
- None. The module imports, runs, round-trips correctly through real JSON, holds no I/O, and its enums match the live schema exactly. Nothing blocks downstream consumption.

### MEDIUM severity
- **U28 round-trip tests can't catch a symmetric dropped field.** `test_session_state_round_trip_stable`, `test_runconfig_round_trip_incl_token_fields`, `test_table_draft_round_trip`, `test_column_draft_round_trip_with_conditionals`, `test_question_round_trip` all assert `x2.to_dict() == x.to_dict()` (tests/test_state.py:94, 106, 115, 130, 162). Because the oracle side is the object's own `to_dict()`, a field omitted from both `to_dict` and `from_dict` round-trips "stably" while silently losing data (demonstrated: `mlflow_run_id="run-XYZ"` → `None` after round-trip, test still green). Since this spine is the persistence contract for the whole core, the tests should pin it harder.
  - Suggested fix unit: add an assertion that compares the round-tripped object's *attributes* (or `dataclasses.asdict`) back to the original, and/or assert the `to_dict()` key set against an explicit expected-keys set, for `SessionState`/`RunConfig`/both drafts/`Question`.
- **No test exercises the `from_dict` defaults for `apply_status` / `status` from an omitting dict.** `test_draft_status_and_apply_status_defaults` (tests/test_state.py:119) checks the *dataclass* constructor default, not the `_draft_common_from_dict` default path. Mutating the `from_dict` default `apply_status` from `"not_applied"` to `"applied"` left all 29 green. A draft dict persisted before the `apply_status` column existed (or any partial blob) would deserialize to the wrong state undetected.
  - Suggested fix unit: add a test that calls `ColumnDraft.from_dict({"column_name": "c"})` / `TableDraft.from_dict({})` and asserts `status == DRAFT` and `apply_status == NOT_APPLIED` (the on-disk default path, distinct from the dataclass default).

### LOW severity
- **`draft_kind` enum parity is unguarded by U28.** The module defines `DraftKind` (table|column) and uses it in `Question.target_kind`; its values match `geniefy.draft_kind` in the migration, but `test_enums_match_migration` (tests/test_state.py:47) parametrizes only the four other enums. If `draft_kind` ever drifts, no test catches it. (Verified manually that it matches today.)
  - Suggested fix: add `(DraftKind, "draft_kind")` to the parity parametrization.
- **`StepResult.ready_for_review` signature differs from the U4 §2 sketch.** LLD §2 writes `ReadyForReview(table_draft, column_drafts, state)`; the module's constructor takes only `state` (drafts already live inside `state`). This is the more correct design (single source of truth, no duplicated drafts) and is internally consistent — flagged only as a documentation-vs-code nuance, not a defect.
- **Minor citation slip in the module docstring.** Line 14 cites "**No I/O, no infra deps (D1)**"; the no-I/O constraint is more precisely D17 (stateless/serializable) — D1 is "core is a UI-free library." Harmless; the intent is correct and D12 (no hardcoded infra) is honored.

### UNVERIFIED
- **Live serialization into the actual Lakebase `jsonb` columns** was not exercised (no DB write in this audit — the tests are hermetic by design). Parity with the schema is proven structurally (enum values + column-name mapping), and U17 already applied the migration live, so confidence is high; an end-to-end `to_dict()` → insert into `sessions.session_state` / `*_drafts` round-trip remains a deploy-time check, consistent with the project's infra-free authoring stance.
