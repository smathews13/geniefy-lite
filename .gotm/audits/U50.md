## Checks performed

**1. Existence + spec/ledger match.** PASS. File exists at `tests/test_providers.py`
(the ledger's stated path, LEDGER.md:76). The unit promises tests for "chat payload
build + response passthrough ¬∑ SQL rows‚Üídicts ¬∑ error propagation"; the file delivers
all three plus the hermetic/lazy-factory dimension named in the dispatch.

**2. Coverage is meaningful.** PASS.
- *Chat payload shape* ‚Äî `test_chat_transport_builds_payload_and_passes_response`
  asserts `endpoint == model`, `payload["messages"]` verbatim, `max_tokens`/`temperature`
  carried, and **`"response_format" not in payload`** when not requested. That omission
  is load-bearing: `LLMClient.complete` always passes `response_format=` as a keyword and
  defaults it to `None` on non-JSON calls (llm.py:189, 204‚Äì205), so the adapter must omit
  it from the wire payload ‚Äî the test guards exactly that real-wiring contract.
  `test_chat_transport_includes_response_format_when_given` covers the present branch.
- *Response passthrough* ‚Äî `test_chat_transport_satisfies_llmclient` drives the adapter
  through a **real** `LLMClient.complete_json` (not a stub): the fake returns the OpenAI
  shape, and the test asserts both the parsed JSON object and `resp.finish_reason ==
  "stop"`, so the adapter's output actually flows through `_parse` (llm.py:208‚Äì230). A
  response-shape drift would fail here.
- *SQL row‚Üídict mapping* ‚Äî `test_sql_runner_maps_rows_to_dicts` (mapping) and
  `test_sql_runner_handles_empty_and_none` (`[]` and `None ‚Üí []`) cover both
  normalization paths.
- *Provider integration* ‚Äî `test_sql_runner_satisfies_lineage_provider` drives a **real**
  `LineageProvider`; the fake `execute` keys on `"SELECT DISTINCT source_table_full_name"`,
  which is verbatim the upstream SQL `LineageProvider.queries` emits (context.py:83) ‚Äî so
  the test exercises the genuine provider‚Üírunner‚Üísnippet path, not a tautology.
- *Error propagation* ‚Äî `test_chat_transport_propagates_errors` and
  `test_sql_runner_propagates_errors` both assert `pytest.raises(RuntimeError)`, matching
  the oracle's division of labor (`LLMClient` owns retries/`LLMError`; `ContextGatherer`
  degrades). Confirmed by mutation probe M1.
- *Lazy SDK factories* ‚Äî covered structurally rather than by a named test: the module
  imports SDK-free (so `from geniefy_app.providers import FmapiChatTransport,
  WarehouseSqlRunner` at line 16 succeeds with no SDK present ‚Äî the whole suite's
  precondition), and the factories `databricks_chat_query` / `databricks_sql_execute`
  lazy-import the SDK inside the function body (providers.py:82, 97), raising
  `ModuleNotFoundError` only when **called** in an SDK-free interpreter. The test does not
  invoke the factories (correct ‚Äî they need live infra and are deploy/integration-verified,
  per the docstring and U49 audit). The "import SDK-free, raise only when called"
  guarantee is satisfied; see Finding (LOW) on directness.
- Mutation-probe verdict: both targeted mutations were caught ‚Üí the suite is not vacuous.

### LOW severity
- **Lazy-factory call-time error is not directly asserted by a named test.** The dispatch
  asks that "the lazy SDK factories are tested to import SDK-free and raise only when
  called." The *import SDK-free* half is exercised (the suite runs SDK-free). The *raise
  only when called* half is verified by this audit (calling `databricks_chat_query()` in
  the SDK-free venv raises `ModuleNotFoundError` at the lazy-import line) and by the U49
  audit, but `test_providers.py` itself has no test that calls a factory and asserts
  `pytest.raises(ModuleNotFoundError)`. Not blocking ‚Äî the guarantee is real and
  structurally enforced (a factory that eagerly imported the SDK would break the import of
  the whole module and fail all 8 tests). A one-line direct test would make the second
  half self-documenting in the suite. Optional follow-on; fold into the existing
  test-hardening sweep if convenient.

  `providers.py` U49 and the test header labels `test_providers.py` U50 ‚Äî each file names
  its own unit correctly per the ledger. Noted only for completeness; no action.
