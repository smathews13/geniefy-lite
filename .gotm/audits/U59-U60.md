## Findings

### HIGH severity

### MEDIUM severity

- **Questions ‚Üí resume ‚Üí re-poll loop does not reliably auto-advance** (`QuestionsPanel.tsx` + `App.tsx:8-9` + `hooks.ts:35-45`, root cause in backend resume): submitting answers (`useAnswers`) POSTs `/answers` (202) and the backend runs `execute_resume` as a *background task*. `useSessionMutation` invalidates `['session', id]` immediately on the 202. But the backend resume path (`orchestrator.resume ‚Üí _redraft ‚Üí _gate`) never persists an intermediate **in-flight** status ‚Äî the session stays `awaiting_input` (set by the prior `_gate`) until the resume completes and flips it straight to `ready_for_review`. Since `awaiting_input` is in App's `DONE` list and **not** in `useSession`'s `IN_FLIGHT`, `refetchInterval` returns `false`, so polling **stops**. Whether the UI ever shows the resumed `ready_for_review` state then depends on a race: if the invalidation-driven refetch lands before the background task flips status (the likely case), the screen stays parked on the QuestionsPanel with no further auto-poll until the user triggers another mutation. U9 ¬ß4/¬ß8 imply the run resumes visibly ("re-reasons just those items"); the current poll gating can leave it stuck. Note: `awaiting_input` correctly keeps ReviewView mounted (no blank screen), so this is degraded-liveness, not a crash.
  - Suggested fix unit: either (a) have the backend persist a transient in-flight status (e.g. `reasoning`) at resume entry so the existing poll latches on, or (b) on `useAnswers` success keep the session query polling for one cycle (treat `awaiting_input` after an answer-submit as in-flight) / set `refetchInterval` to also poll briefly after a resume mutation.

### LOW severity

- **Unhandled `paused` / `cancelled` statuses render a blank main area** (`App.tsx:82-90`): `SessionStatus` has 11 members; App routes `IN_FLIGHT` (5) ‚Üí RunView, `DONE` (3: `awaiting_input`/`ready_for_review`/`applied`) ‚Üí ReviewView, and `failed` ‚Üí error card. `paused` and `cancelled` match none of the branches, so the `<main>` shows nothing (only the header + "New run"). Not reachable in the happy demo path, but a real terminal/blank gap.
  - Suggested fix unit: add a neutral terminal card for `paused`/`cancelled` (or fold into the failed/terminal branch).

- **Bundle size 569.95 kB (>500 kB warning)** (vite build): recharts (`JudgeRadar`) dominates. Already tracked as the deferred code-split sweep in the ledger. Cosmetic/perf only.

- **`ResultsView.tsx` is orphaned-but-harmless** (per the audit brief): the superseded read-only U59 drafts view; App now routes terminal states to `ReviewView`. It still compiles/typechecks and is exported nowhere consumed. No action required; could be deleted in a cleanup unit. (Noted, not failed.)

- **Apply errors are not surfaced in `ReviewView`** (`ReviewView.tsx:39-46`): `useApply` exposes `isError`, but the button only reflects `isPending`; a failed apply (or the 501 when `applier` is `None` in non-wired contexts) produces no user-visible message. In the real wiring path `applier=Applier(sql, store)` is injected (`api.py:307`), so 501 is test-only ‚Äî hence LOW, not a flow break. U6 ¬ß5/¬ß8 want clear per-item conflict/error surfacing; per-item apply-status badges exist in `DraftCard` (`APPLY_BADGE`) but a top-level apply failure is silent.
  - Suggested fix unit: render `apply.isError` (and any returned per-item `conflict`/`failed` from the apply response) as an inline notice in ReviewView.

- **Edit-in-place text can go stale after resume/re-poll** (`DraftCard.tsx:43`): `useState(draft.proposed_comment ?? '')` seeds once; if a draft's `proposed_comment` changes server-side (e.g. after a resume re-draft) the editor `text` won't resync until remount. Minor: `editing` closes on save success, and reopening Edit shows the original (now-stale) seed rather than the new proposed text. Demo-tolerable.
