### LOW severity
- **Detail-view double Cancel possible:** `SchemaRunDetail` disables the Cancel button only on `cancel.isPending`; after a successful cancel the run row may still read `running` for up to one poll cycle (≤1.5s) until the next `useSchemaRun` refetch lands, leaving the button briefly clickable again. Harmless — the backend cancel is best-effort/idempotent and `useCancelSchemaRun` invalidates the query on success — but a momentary second click is technically possible. `SchemaRunView.tsx:130-139`.
- **Unused `qc` invalidate breadth on cancel:** `useCancelSchemaRun` invalidates both `['schema-runs']` and the *prefix* `['schema-run']` (`hooks.ts:188-191`). TanStack treats `['schema-run']` as a prefix match so it does invalidate the active `['schema-run', runId]` query — correct — but it is broader than necessary. Cosmetic.

### UNVERIFIED
- **Live in-browser visual verification** (does the tab render, do chips/poll/cancel behave against a real backend, do table rows actually navigate into the review/questions/apply screens on click): explicitly OUT OF SCOPE for this audit and deferred to U113 (live deploy + verify). Static wiring is verified (see Wiring below); pixel/runtime behavior is not.

### Contract match — PASS
- **Routes/methods exact:**

  - `useSchemaRun` → `GET /api/schema-runs/{runId}` ↔ `api.py:461`. ✓
  - `useStartSchemaRun` → `POST /api/schema-runs` ↔ `api.py:447`. ✓
  - `useCancelSchemaRun` → `POST /api/schema-runs/{runId}/cancel` ↔ `api.py:468`. ✓
- **Start body `{catalog, schema, filters}`:** `hooks.ts:177` posts `{catalog, schema, filters}`; backend `SchemaRunRequest` (`api.py:441-445`) declares `catalog`, `schema_name = Field(alias="schema")`, `filters` with `populate_by_name=True` — so the client's `"schema"` key binds via the alias. ✓ The view sends `filters: { skip_documented: skipDocumented }` (`SchemaRunView.tsx:50`); `filters` is `dict | None` server-side, so any shape is accepted. ✓
- **Response wrappers:** list returns `{schema_runs: [...]}` (`api.py:459`), hook types it `{schema_runs: SchemaRun[]}` and reads `data?.schema_runs ?? []`. ✓ Detail returns the bare run dict (`api.py:466`), hook types it `SchemaRun`. ✓ Start returns `{schema_run_id}` (service `start_schema_run`), hook types `{schema_run_id: string}` and the view reads `r.schema_run_id`. ✓
- **`SchemaRun` field parity:** `_schema_run_row` (`store.py:550-554`) emits `id, catalog, schema, status, filters, total_tables, counts, job_run_id, created_by, created_at, updated_at`; TS `SchemaRun` (`types.ts:126-139`) declares exactly these + optional `sessions?` (attached only by `get_schema_run` at `api.py`/service `get_schema_run` via `list_sessions`). ✓ Nullability matches (`total_tables: number | null`, `job_run_id: number | null`, `created_at/updated_at: string | null`).
- **`SchemaRunStatus` values:** TS = `enumerating | running | completed | completed_with_errors | failed | cancelled` (`types.ts:119-120`). Backend uses these verbatim — driver finalizes `completed`/`completed_with_errors`/`failed` (`schema_run.py:49-55`), service sets `running`/`cancelled` (`api.py:252`, store `finalize_schema_run` cancel), enum default `enumerating`. All six match; the `RUN_BADGE` Record is keyed by all six (`SchemaRunView.tsx:6-13`) so `RUN_BADGE[r.status]` is total (no undefined-class crash). ✓
- **`SchemaRunCounts` keys:** TS optional `ready/needs_input/applied/error/skipped` (`types.ts:122-124`) ↔ backend allowlist `_SCHEMA_RUN_COUNT_KEYS = {"ready","needs_input","applied","error","skipped"}` (`store.py:38`) and driver bump keys (`schema_run.py:19,45,49,52`). Exact set match. ✓
- **`SessionSummary` (run-detail rows):** `_session_summary` row (`store.py:243-245`) emits `session_id, target, status, created_by, created_at, updated_at, n_columns, n_applied`; TS `SessionSummary` (`types.ts:91-100`) matches. The view reads `s.session_id`, `s.target`, `s.status`, `s.n_applied` — all present. ✓
- **Poll gating:** `useSchemaRun.refetchInterval` returns `POLL_MS` only when `status === 'enumerating' || status === 'running'`, else `false` (`hooks.ts:166-170`) — matches the LLD §7 "polls while `status ∈ {enumerating, running}`" requirement and stops on every terminal status. ✓
