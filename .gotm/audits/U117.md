### MEDIUM severity
- **`--use-job` migrate path is NOT branch-corrected — the U117 bug persists on the alternate path.** Step 5's job arm (`deploy.sh:171-173`) runs `bundle run geniefy_setup -t "$TARGET"` and does **not** pass `GENIEFY_LAKEBASE_BRANCH`. The `geniefy_setup` job in `databricks.yml:100-113` sets no branch env on the task, so `migrations/run_migrations.py` falls back to its default `BRANCH = "production"` (`run_migrations.py:32`). Therefore `./deploy.sh -t dev --use-job` would still migrate the `production` branch — the exact mismatch U117 set out to fix — leaving the dev app's `dev` branch un-migrated. Not HIGH because the local runner is the default (`USE_JOB=0`) and U107-validated path, the U117 row explicitly scopes the fix to "deploy.sh's migrate (step 5) + endpoint resolution" (step 5's local arm IS fixed), and a complete fix touches `databricks.yml` (outside U117's stated target). The existing/working dev deploy uses the local path and is unaffected.
  - Suggested fix unit: parameterize `geniefy_setup` with `GENIEFY_LAKEBASE_BRANCH` (env in `databricks.yml` keyed off `${bundle.target}`, or pass it on the `bundle run`), so `--use-job` tracks the target too.

### LOW severity
- **Branch-just-created → wait loop assumes the new endpoint appears immediately.** When the `dev` branch does not exist, the script `create-branch` + `create-endpoint` (lines 127–132, both best-effort `|| log`), then immediately enters the ACTIVE-wait loop (lines 135–142) polling `list-endpoints .../branches/${DB_BRANCH}`. If endpoint creation is async and the endpoint isn't yet listable on the first iterations, `jq '.[0]...'` yields `PENDING` (the `// "PENDING"` default) and the loop simply keeps polling for up to 60×5s — correct, self-healing behavior. Only flagged because a hard create-endpoint failure degrades silently to a log and the script would then spin the full 5 min before `die`-ing with "did not reach ACTIVE"; acceptable for an idempotent re-runnable script. No action required.
