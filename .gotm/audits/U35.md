### LOW severity
- **`summarize` compaction call is itself unbudgeted (D32 / U24 §2.1).** `_summarize` (reasoner.py:198-210) calls `self.llm.complete(...)` with no `max_input_tokens`, so the cheap summarization call — which D32 §2.1 lists as a budgeted call type ("`summarize` (the compaction call itself)") — can exceed its own input budget on a pathologically large context. Impact is bounded: it is invoked only as a reducer when the main prompt is already over budget, it caps output via `max_tokens=200`, and on any `LLMError` it falls back to a truncated join (line 209-210) rather than failing the run. Honest cost-predictability nit, not a correctness defect. Suggested fix unit: thread a `summary` input budget (`RunConfig.summary_target_tokens` / a per-phase budget) into the `_summarize` call.
- **"Shrink the batch" reducer (U24 §2.3 step 4) not wired into the Reasoner pipeline.** The compaction pipeline in `_complete` (reasoner.py:182-187) registers three reducers — `trim_context → summarize_context → reduce_profile` — but not the design's 4th ordered step, "shrink the batch — fewer columns per `reason` call, re-batching the remainder" (NFR-plan §2.3 step 4). If a single over-budget batch is still over after step 3, the prompt is sent over budget (Budgeter returns `within_budget=False`) rather than re-batched. The module docstring (lines 16-18) honestly enumerates only the three reducers it implements, so this is an under-implementation of the sketched 4th step, not a misrepresentation; the `state.py` `reason_batch_size` is the static bound. Acceptable for v1 (the design marks the truncation-marker fallback as the floor), but worth tracking. Suggested fix unit: add a re-batching reducer (or honest `within_budget=False` → draft `warning`) in a follow-on, when the Orchestrator wires per-batch budgets.
- **Profile `existing_comment` lookups use `dict.get` without guarding non-dict batch entries.** `_draft_columns` (reasoner.py:108-109,134-136) and `_draft_table` assume each column/table block is a dict; a malformed profile (non-dict column entry) would raise `AttributeError` mid-batch rather than flagging that batch `error`. Low because the profile is produced by the audited Profiler (U14) to the U3 schema, so this is a hostile-input edge, not a realistic path. Suggested fix unit: defensive coercion if the Reasoner is ever exposed to unsanitized profiles.

### UNVERIFIED
- **MLflow tracing spans (U4 §6 — `reason` span per batch).** The Reasoner emits no MLflow spans; §6 says "Spans: … `reason` (one per batch)". This is consistent with the project's deferral pattern (tracing is wired by the Orchestrator/app layer, not the leaf modules — `llm.py` and `judge.py` likewise emit none) and the hermetic constraint (D1: no I/O in the core leaf modules), so it is almost certainly out of scope for U35. Flagged only because the spec text attaches the span to "reason" — confirm tracing is the Orchestrator's job, not the Reasoner's.
