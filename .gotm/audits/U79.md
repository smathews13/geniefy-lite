### LOW severity

- **E1 "20K-token comment" framing slightly overstates the actual cap.** §E1 raises the *template word caps* (table 500 words, column 150 words) but the *output-token budget* (`reason_table_max_tokens=20000`) governs the whole model call (the table draft JSON: `proposed_comment` + `rationale` + `evidence_refs` + …), not the comment alone. A 500-word comment is ≈650–700 tokens, far under 20K; the 20K budget is headroom for the full structured response across the batch, which is correct, but the audit-prompt's shorthand "a 20K-token comment" could mislead a reader of the design into thinking comments themselves may reach 20K tokens. The honest UC-length caveat is present (§E1 bullet 3 + D43: "Applier truncates-with-warning if a comment exceeds the UC length limit"), so the substance is covered; only the token/word framing is loose. Cosmetic — fix if convenient when U81 lands.

- **§E1 "Implementing: U81 → builds on U80's budget plumbing" overstates the U80→U81 dependency.** U80 (E8) is LLM-retry/backoff/jitter/429 work; it does *not* introduce the per-phase token budgets that U81 adds (those are new `RunConfig` fields). The phrase reads as if U80 lays "budget plumbing" U81 extends, but U80 touches `llm.py` retry only. The ledger's U81 inputs (`U79, U25, U35, U43`) correctly omit U80, so the dependency is not actually wired wrong — only the prose is slightly misleading. Cosmetic.
