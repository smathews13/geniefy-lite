## Findings

### HIGH severity

- **D25 violation — `databricks.yml` declares no `sync: exclude:` for `.gotm/` and dev `.claude/`** (`databricks.yml`, whole file; the key is absent — confirmed by `grep -niE "exclude|\.gotm|\.claude|sync:"` → no match). D25 is explicit and dual-sourced: the decision (`.gotm/DECISIONS.md:343-353`, esp. line 351) and the U8 oracle (`docs/design/LLD-packaging-deployment.md:89-92`, §9 "What ships vs. dev-only") both state that `.gotm/` and dev `.claude/` are kept in git as build provenance **but excluded from the `bundle sync` deploy artifact so they are never uploaded to the workspace.** A `.claude/` directory is present in the repo (`.claude/settings.json`, `.claude/settings.local.json`) and a populated `.gotm/` tree exists, so with no `sync.exclude` block a `databricks bundle deploy` would sync GOTM orchestration + the dev immutability hook into the customer/FE-VM workspace — exactly the leak D25 forbids. This is the named CRITICAL check for this audit and a do-not-deploy-correctly issue (GOTM build machinery leaking into a customer workspace), so it is HIGH. The DAB is otherwise well-formed and would `validate`, but it does **not** honor the recorded packaging decision it was built to implement.
  - Suggested fix unit: append **U15 — "Fix(U13): add `sync.exclude` to `databricks.yml` (`.gotm/`, `.claude/`, `**/.gotm/**`, `**/.claude/**`) per D25/U8 §9"**; references `.gotm/audits/U13.md`. (Do not edit the frozen U13 output in place beyond what the no-edit-done rule allows — land the change as the follow-on unit's output; since `databricks.yml` is U13's frozen output, the fix unit supersedes that file.)

### MEDIUM severity

- **`prod` target has no `workspace.host` and no `${var.host}` variable, diverging from the U8 illustrative DAB** (`databricks.yml:37-40` vs `docs/design/LLD-packaging-deployment.md:49-51`). U8 §3 models both targets with `workspace: { host: "${var.host}" }` and §4 lists `host` as a first-class bundle variable. The scaffold instead hardcodes the `dev` host (line 34) and leaves `prod` with only a comment ("Customer sets host via their own CLI profile / --profile"). This is defensible (D28 says `dev` carries FE-VM defaults and `prod` stays parameterized; a CLI-profile host is a legitimate DAB pattern) and keeps `prod` parameterized — so it does not break the D12 "prod stays parameterized" requirement — but it is an undocumented departure from the U8 design's `host`-as-variable shape, and `databricks bundle validate -t prod` with no profile/host resolves to no host. Consumable, but the deploy unit should confirm the customer-host story matches the README prerequisites. Not blocking.
  - Note for the deploy/README unit: either add a `host` variable (U8 shape) or document the `--profile`/`-p` host story for `prod` in the customer-facing prerequisites.

- **README's design-doc links omit the U8 packaging/deployment LLD and several shipped LLDs** (`README.md:33-36`). The "Design docs" list names only `HLD.md` and `LLD-data-layer.md`, then says "further `LLD-*.md` added per component." All links that are present resolve (verified on disk), but for a customer-facing OSS README that points readers at the design set, omitting `LLD-packaging-deployment.md` — the very doc that documents how to deploy this scaffold — is a quality gap. D12/U8 §5 also promise a customer-facing **prerequisites + deploy steps** section (Lakebase enabled, FMAPI Claude + AI Gateway, SQL warehouse, the `bundle deploy` commands); the README has none — it covers mission/why/organization/license but not "how to deploy," which the unit's product-README intent (and U8 §5) calls for. Non-blocking for the scaffold's validity, but the deploy unit / a README follow-on should add a Prerequisites + Deploy section and link the packaging LLD.
  - Note for the deploy unit: add Prerequisites + `databricks bundle deploy` steps to `README.md` (living governance doc — editable) per U8 §5 / D12.

- **`migrations/`, `src/geniefy_core/`, `resources/`, `profiling/` from the U8 ship list (§2) are not present in this scaffold** (`docs/design/LLD-packaging-deployment.md:22-34`). This is correct and consistent with the unit's intent — U13 is the scaffold + DAB + README + LICENSE only; `migrations/001_init.sql` is U12 (already done) and `src/geniefy_core/` is U14+ — and `databricks.yml:48-50` explicitly defers resource bindings (jobs, db instance, secret scope, warehouse/endpoint bindings) to later units "so the scaffold validates cleanly." Recorded as LOW/informational so a later reader does not mistake the partial tree for a spec miss; it matches the U8 phased build, not contradicts it.

Static read suggests it would validate for the `dev` target: `bundle.name` present (`databricks.yml:7`), well-formed `variables` (lines 9-27), `targets.dev`/`targets.prod` (lines 29-40), and `resources.apps.geniefy` with `source_code_path: ./app` (lines 42-50).
