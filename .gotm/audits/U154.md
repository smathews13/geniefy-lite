## Method
Read the target doc, the LEDGER U154 row, D59 (the ratified decision), and the cited guardrails

## Findings

### HIGH severity
- *(none)* — the design is buildable as written. All data the rollup + bucket logic needs is present
  client-side, a per-draft approve path exists server-side to base the bulk endpoint on, the gate's
  status routing exactly matches the doc's selection rule, and no guardrail is contradicted.

### MEDIUM severity
- **Bulk-approve audit actor is misattributed to "request identity (D48)" — contradicts the D48 split.**
  §3 "Server endpoint" says the endpoint "writes the `audit_log` `approved` rows (actor = the request
  identity, D48)". But D48's locked refinement (`DECISIONS.md` D48, codified in `api.py:382-385` `_actor()`:
  *"run/resume/**review** all run as the SP"*) reserves the **end-user/OBO identity for the apply write
  only**; every other route — including review/approve — is attributed to the **app service principal**.
  Bulk-approve is a review action, so its `audit_log.actor` should be the SP, not the OBO end user. As
  written, a build unit that reads "request identity" as the `X-Forwarded-*`/OBO user would violate D48.
  The doc's *intent* is correct (bulk-approve ≠ apply, no OBO, no UC write), but the parenthetical
  mislabels D48. (Lower-bounded MEDIUM: the surrounding text is unambiguous that bulk-approve never
  applies; only the actor attribution is wrong.)
  - Suggested fix unit: amendment correcting §3 to "actor = the app SP (D48: only apply uses OBO/end-user)".

- **"Reuses the existing per-draft review path's approve semantics" overstates what exists for the audit row.**
  §3 claims the endpoint "writes the `audit_log` `approved` rows … (Reuses the existing per-draft review
  path's approve semantics.)" The existing per-draft approve (`api.py:review` → `SessionService.review_draft`,
  `api.py:150-173`) sets `status=APPROVED` and best-effort upserts the comment library, but it does **not**
  write any `audit_log` row — only the apply path calls `append_audit` (`api.py:527`; `store.append_audit`
  at `store.py:174` is invoked solely by the applier). So the audit-row behavior the bulk endpoint describes
  is **net-new**, not reused. The helper (`append_audit`) exists and is callable, so this is buildable — but
  the design should not present the audit-row write as pre-existing review semantics, and a build unit
  should be told to add an approve-audit row (and decide whether per-draft approve gains one too, for parity).
  - Suggested fix unit: clarify §3 — the bulk endpoint adds a *new* `approved` audit row via `append_audit`; per-draft approve currently writes none.

### LOW severity
- **Bulk-approve status filter is stated two slightly different ways; reconcile the prose.** §3 first lists
  the rule as "`confidence ≥ keep_threshold` AND not flagged: `status == 'draft'` (NOT
  needs_input/low_confidence/error, and not already approved/edited/applied)". The `status == 'draft'` clause
  is correct and matches the gate exactly (`gate.py:123-125`: a *kept* high-confidence, no-trip draft is
  "left as-is", i.e. stays `DraftStatus.DRAFT`), and the threshold check is then redundant-but-harmless
  (a below-threshold or tripped draft is already `needs_input`/`low_confidence`, never `draft`). This is
  internally consistent; the only nit is that listing both the threshold AND `status=='draft'` could read
  as two independent gates when in practice `status=='draft'` alone is sufficient post-gate. Recommend the
  build LLD note that `status=='draft'` is the operative filter and the threshold re-check is a belt-and-suspenders
  guard (and clarify behavior if `keep_threshold` is changed *after* a gate run — drafts keep their gated status).
- **`confidence`-rollup band/threshold coupling unspecified.** The doc renders the aggregate "ConfidenceChip-styled".
  `viz/confidence.ts` bands at fixed cutoffs (≥0.8 high, ≥0.6 mid, else low) that are **independent** of the
  configurable `keep_threshold` used for the bucket/selection logic. That's fine (color band ≠ review-ready
  bar), but a build unit could conflate them; a one-line note that the meter's color band and the
  review-ready bucket use different thresholds would prevent a wrong implementation.

### UNVERIFIED
- *(none material)* — all five points were checkable against code + decisions.

   and `get_schema_run` attaches those summaries, not full drafts (`api.py:315`). The doc proposes a
   **server-computed `confidence_summary` per session** to fill this, which is the right call (no such field
   exists today — confirmed). No assumed-but-absent endpoint/data otherwise.

5. **Honesty / governance invariants** — PASS. §5 holds all five: truthful confidence (always paired with
   breakdown + weakest caveat), distinct meters, no blanket approve, bulk-approve ≠ apply, schema-level
   bulk-approve writes nothing to UC. Consistent with D22/D23 and D7/D48.

## Design-soundness assessment
- **Weighted formula** `0.5*table + 0.5*mean(cols)` is reasonable and honest: anchoring half-weight on the
  table headline + half on the column mean is defensible and **paired with the weakest caveat** so a high
  mean can't mask one bad column (the honesty risk a pure mean would carry). Acceptable as ratified in D59.
- **Null-confidence handling IS specified** (§2): null drafts excluded from the mean; all-null → `overall = null`
  → render "—" (matches `pct(null)` → "—" in `confidence.ts:33`). Good.
- **"review-ready" is defined consistently with the Gate** (§2): `confidence ≥ keep_threshold` AND not flagged
  — which equals the gate's "kept, left as DRAFT" set (`gate.py:123-125`), so the bucket count and the
  bulk-approve target are the same population. Internally coherent.
- **Ambiguity a build unit could trip on:** the two MEDIUM items (audit actor; "reused" audit semantics) and
  the LOW band-vs-threshold coupling. None block the build; all are addressable in the build LLD or a small
  amendment.
