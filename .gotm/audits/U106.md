## Findings

### HIGH severity
None.

- **Build/typecheck:** `npm run build` passes clean (tsc + vite). The SPA ships.
- **Contract match (frontend ↔ backend) — verified exact:**
  - `useLibrary` → `GET /api/library`; query params `scope`/`limit`/`offset` plus `include_sunset` emitted **only when truthy** as the string `'true'` (`hooks.ts:63-64`). Matches `api.py:312-317` (`list_library(scope, include_sunset, limit, offset)`) and FastAPI's bool coercion of `?include_sunset=true`.
  - `useSunsetLibrary` → `POST /api/library/{id}/sunset` (`hooks.ts:75`); `useReviveLibrary` → `POST /api/library/{id}/revive` (`hooks.ts:84`). Exact match to `api.py:319` / `api.py:329`. `id` is URL-encoded.
  - `LibraryEntry` TS fields (`id, scope, match_key, canonical_comment, tags, usage_count, source_table_ref, approved_by, updated_at, status`, `types.ts:105-116`) match the backend `store.list_library` SELECT/row dict **field-for-field** (`store.py:377-391`).
  - `LibraryStatus = 'approved' | 'applied' | 'sunset'` (`types.ts:103`) matches the backend lifecycle values (D52; `store.py` writes `'approved'`/`'applied'`/`'sunset'`).
  - `DraftBase.tags: string[]` (`types.ts:29`) matches the backend draft `tags` emitted by `_draft_common_to_dict` (`state.py:261`, shared by Table + Column drafts) and the `TableDraft.tags` / `ColumnDraft.tags` `list[str]` fields (`state.py:195,224`).
  - `ColumnDraft` TS (`column_name, ordinal, data_type, conditional_fields`, `types.ts:39-44`) matches `ColumnDraft.to_dict` (`state.py:234-237`). `TABLE_TARGET='__table__'` (`types.ts:137`) matches the backend target sentinel (`api.py:152`).
- **No crash on empty tags:** `PillRow` returns `null` on null/empty arrays (`Pill.tsx:48`) — no empty container, no crash. Both card variants pass `draft.tags ?? []` (`DraftCard.tsx:70`) and LibraryView passes `e.tags` (backend always returns a list via `_parse_tags`).
- **Unknown free-form tags render (D53/Q2):** `Pill` tone='tag' falls back to `DEFAULT_HUE` (slate) for any tag not in `TAG_HUES` (`Pill.tsx:33`) — free-form tags like `revenue`/`arpdau` render, never break. PII/`sensitive`/`sensitivity` get the rose alert hue (`Pill.tsx:6-8`).

### MEDIUM severity
- **§B4 / U106-row "steward facts row" not rendered as a discrete scannable row** (`DraftCard.tsx:95-120`): the U106 ledger row promises a *"steward facts row owner/freshness/grain/keys/sensitivity"* and LLD §B4 calls for a *"scannable, icon-keyed steward facts row (owner · freshness/SLA · grain · keys · sensitivity)."* The implemented hero renders the eyebrow ("Table · data steward view") + a prominent FQN + the trust ConfidenceChip + status + table-tag pills + the richer comment **body** (rendered as prose with `whitespace-pre-line`, `:175`). It does **not** render those five facts as discrete, icon-keyed cells. Root cause is upstream: the backend `TableDraft` surfaces the comment only as the single free-text `proposed_comment` blob (`state.py:188-200`) — by design B1 generates the steward facts as prose *inside* the comment, and `SessionView` exposes no discrete `owner`/`freshness`/`grain`/`keys`/`sensitivity` fields and no parsed structure. So the facts *are present* (in the comment prose, which the hero shows), but not as the promised scannable row. Non-blocking: the hero is consumable, visually dominant, and steward-oriented; this is a fidelity gap on one sub-bullet of §B4 / the row title, not a contract/build/regression failure.
  - Suggested fix unit: either (a) surface structured table fields from the backend (template `conditional_fields` → `SessionView`) and render an icon-keyed facts row in the hero, or (b) ratify (DECISIONS/QUESTIONS) that steward facts live in the comment prose and amend the U106-row/§B4 wording to match. Defer the visual judgment to U107 (live verify).

### LOW severity
- **Column-card data-type pill omitted when `data_type` is null** (`DraftCard.tsx:126`): `{subtitle && <Pill tone="type">…</Pill>}` — correct/defensive (no empty pill), but a column with no `data_type` shows no type chip at all. Matches §B3 intent (type → pill) for the common case; acceptable.
- **`Pill` accepts only `string` children** (`Pill.tsx:27`) and coerces via `String(children)` (`:31`); fine for the current callers (all pass strings) but the `String()` coercion is dead for a `string`-typed prop. Cosmetic.
