### LOW severity
- **`append_audit` lacks the defensive rollback the other writes have** (`store.py:174-184`): `save` (98-106) and `upsert_library_entry` (247-280) both wrap their write+commit in `try/except` with a nested-try defensive rollback (so a rollback failure can't mask the real error). `append_audit` does the INSERT + `conn.commit()` with no surrounding try/except, so on a write error it neither rolls back nor re-raises a wrapped error. Impact is bounded and non-blocking: on the factory (production) path the per-call conn is closed unconditionally by `_connection()`'s `finally` regardless, so no stale/aborted conn leaks; on the static test path `_connection()` rolls back at the *start* of the next call, so an aborted txn self-heals before reuse. D49's "writes commit on success; on error they roll back defensively" is thus satisfied in effect for the connection-lifecycle (the audited concern) but `append_audit` is stylistically inconsistent with its sibling writes. Cosmetic / defensive-hardening only — does not affect the per-call/no-cache contract.
- **Doc-comment for `list_sessions` says "reuses the shared connection"** (`store.py:208`): a stale phrase from the pre-D49 single-conn design. The method in fact goes through `_connection()` (line 212), which is per-call under a factory. Cosmetic docstring drift; behavior is correct.

- None. All checks were mechanically verifiable from source + the hermetic suite.

## Check-by-check

Every public op goes through `with self._connection() as conn`: save (97), append_audit (174), load (191), list_sessions (212), upsert_library_entry (246), list_library (288). No op bypasses `_connection()` or references `self._conn` outside the static-fallback branch.
