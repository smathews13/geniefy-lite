### LOW severity
- **OBO Statement Execution host source is deferred without explicit naming of the resolver (A5, line 73).** The doc says the user-token apply client uses base path `{host}/api/2.0/sql/statements` but does not say where `{host}` comes from at request time. The U76/U77 work established that the app self-OAuths and reads its host from the Apps environment (`DATABRICKS_HOST`/SDK config), and the existing `WarehouseSqlRunner` (U49) already resolves a host; A5 says it "reuses the existing `WarehouseSqlRunner`/type-coercion path … only the credential changes," which implicitly inherits that host. This is consistent but a code unit (U85) implementer would benefit from one explicit sentence stating the host is the app's own workspace host (same as the SP path), since OBO swaps only the bearer token, not the workspace. Non-blocking — the "reuse the existing path, change only the credential" framing makes the intent recoverable.
- **`CAN_USE` on the warehouse is asserted as generally-true rather than verified (A5, line 74: "the user must have `CAN_USE` on it (they generally do)").** This is a reasonable design assumption (warehouse-level `CAN_USE` is broad and separate from catalog `MODIFY`), and the degrade-not-silent-SP fallback (A5 fallback 1) plus the `PERMISSION_DENIED` surfacing (fallback 2) would catch a missing-`CAN_USE` case as a per-item error rather than a silent failure, so it is not a correctness gap. Worth a one-line note in U85 that a warehouse-permission denial surfaces through the same per-item error channel as a catalog denial.

- None. All oracle sources (D48, §E9) and cross-referenced units (U91/U84/U85/U53/U55/U49 in LEDGER) were present and readable; header ground truth is well-established and required no external fetch.
