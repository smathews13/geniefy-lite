## Findings

### HIGH severity
*(none)*

The five-point check passes:
1. **existence** — both files exist at the ledger's stated paths.
2. **spec match** — `profile_table` returns the U3 §4.2 shape: a `table` block (`full_name`, `row_count`, `row_count_is_estimate`, `sampled`, `column_count`, `partition_columns`, `existing_comment`, `stats_source`, `profiled_at`, `warnings`) and `columns` with `type_class` / `null_fraction` / `distinct_count` (+ `distinct_is_approx`) / `cardinality_ratio` / `is_enum_candidate` / `top_k` / `min`-`max`. `profile_schema_version` is pinned to `"1.0"`, matching the core's `PROFILE_SCHEMA_VERSION` floor (profiler.py:39-40).
3. **cross-reference integrity** — D25 (in-app provider, pluggable), D4 (sanitized-at-source), D15 (provider-agnostic) all exist and say what the docstring claims.
4. **internal consistency** — type classification, enum gating (`0 < ndv <= max_card`), and top-K emission are mutually consistent; the end-to-end test wires the provider through the real `geniefy_core.profiler.Profiler` and the `ProfileProvider` protocol conforms (`runtime_checkable`, single `profile_table(request) -> Mapping` method — satisfied).
5. **decision fidelity (D4 — PII sanitized at source)** — **confirmed.** Classification is by column name (`_NAME_PII`) **and** by value pattern (`_VALUE_PII`); for any classified column the `top_k` values are masked via `_mask` before return and `pii.action="masked"` is recorded. Raw identifiers are never returned. Independently verified with an adversarial probe: a column **named `code`** (no name match) but holding SSN-shaped values is detected (`classes:['ssn']`) and its top-K values are masked (`12•••••••••`). Hermetic: SQL runner injected (`SqlExecute`), tests drive canned rows. SQL identifiers backtick-escaped (`_ident`, doubles embedded backticks) and literals single-quote-escaped (`_lit`) — adversarial probe confirmed `` `a``b` `` and `s''ch` in emitted SQL.

**Meaningfulness of the suite (would real defects be caught?):** Yes for the three named risks. Un-masked PII → `test_pii_detected_by_name_and_value_and_masked` asserts every top-K value contains `•` and none matches the SSN regex; a regression that emitted raw values would fail. Wrong `cardinality_ratio` → `test_integer_key_column_stats` pins `1.0` for the key column and `test_provider_drives_the_core_profiler` re-checks it through the core. Missing enum-candidate → `test_enum_column_gets_top_k` asserts `is_enum_candidate is True` plus the exact top-K ordering, and `test_integer_key_column_stats` asserts `False` for the 100-distinct key (boundary `100 > 50`). Non-PII values left untouched is also pinned (`test_non_pii_values_not_masked`).

### MEDIUM severity
- **Empty-table `warnings` entry not emitted (U3 §9, LLD line 179).** The §9 error/edge table prescribes that an empty table (0 rows) return `row_count=0`, structure present, distribution fields null, **and `warnings:["empty table"]`**. The implementation handles the numerics safely — for `total=0` it returns `null_fraction=None`, `cardinality_ratio=None`, `is_enum_candidate=False` with no zero-division and no crash (verified by probe) — but it does **not** append the `"empty table"` warning. Downstream the Judge/Gate (D8) keys confidence off `warnings`, so a missing signal here is a quality gap, not a correctness break (the profile is still valid and consumable). `profile_table` at profiling.py:108-113 only appends `sampled` / `no columns found` warnings.
  - Suggested fix unit: append a follow-on unit to add `if row_count == 0: warnings.append("empty table")` and a test asserting it.

No action.
