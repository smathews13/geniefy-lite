## Findings

### HIGH severity
*(none)*

The two load-bearing invariants the spec demanded are preserved:

- **Q7 — no autonomous UC writes (verified, clean).** Every reference to apply in the doc routes through the human/OBO path: §1 ("every apply stays human-in-the-loop via OBO"), the §2 diagram and §3 table ("UC apply … **Never** — run ends at persisted drafts"), §5 ("The **only** UC writer remains the human via OBO at `/api/sessions/{id}/apply` … Hands-off never holds or uses a user token"), §6 entrypoint comment ("generate + persist, NO apply"), §8 scope-guard ("No autonomous apply / no scheduled re-runs"). There is no code path, route, or Job step in the doc that writes to UC. No auto-apply path exists to flag.
- **Q8 — deferred clarification reuses the EXISTING resume path (verified, clean).** §1 and §3 state the hands-off pass persists the same `SessionState` an interactive run produces *at the pause point* (questions attached, `suggested_answer` filled per D50), then stops; later answers go through `SessionService.execute_resume` (U52) → `orchestrator.resume(state, answers)`. I confirmed in `src/geniefy_app/api.py:113-120` that `execute_resume` exists and delegates straight to `orch.resume(...)` — the D17 stateless path — so the doc's "Zero new resume code" claim is faithful and no new resume engine is required or implied.

### MEDIUM severity

- **M1 — `RunMode.HANDS_OFF` names a type that does not exist; the real seam is `SessionMode` and `RunConfig.mode`.** §3 (`docs/design/LLD-amend-005.md:46`) introduces "a new **`RunMode.HANDS_OFF`** … selected via `RunConfig.mode`." But the codebase/frozen design has **no `RunMode`** — the enum is `SessionMode` (`src/geniefy_core/state.py:30`, values `interactive`/`batch`) and `RunConfig.mode: SessionMode` (`state.py:117`). The Postgres mirror is `geniefy.session_mode AS ENUM ('interactive','batch')` (`migrations/001_init.sql:15`). The design is buildable (add a third enum member), but a build unit following the doc literally would invent a wrong type name. Build units must add `hands_off` to `SessionMode` (and, if the per-table session's mode is persisted to `sessions.mode`, to the `session_mode` Postgres enum — see M2), not create a parallel `RunMode`.
  - Suggested fix unit: append a follow-on note/amendment correcting "`RunMode`" → "`SessionMode`" and pinning the enum-extension surface (Python `SessionMode` + `session_mode` PG type).

- **M2 — migration `003` omits the `session_mode` enum extension that the new mode implies, and drops the schema-qualification / search_path conventions of `001`.** §4's `003_schema_runs.sql` (`docs/design/LLD-amend-005.md:63-79`) adds `schema_runs` + `sessions.schema_run_id` but does **not** add a `'hands_off'` value to `geniefy.session_mode`. If per-table sessions created by the Job record their mode (the existing `sessions.mode NOT NULL DEFAULT 'interactive'` column, `001_init.sql:104`), the batch sessions must be writable as a hands-off mode — either by extending the enum (`ALTER TYPE geniefy.session_mode ADD VALUE 'hands_off'`) or by an explicit decision to store them as `'batch'`. The doc is silent. Separately, the migration SQL is unqualified (`CREATE TABLE schema_runs …`, `ALTER TABLE sessions …`) and omits the `set search_path to geniefy` / `geniefy.` schema prefix that `001_init.sql` uses throughout (`001_init.sql:8-9`, all `geniefy.*` objects) — applied as-is it would land in the wrong schema. Coherent intent, but the build unit must reconcile schema qualification and the mode-enum gap.
  - Suggested fix unit: in the data build unit, qualify `003` to `geniefy.*` + `set search_path`, and decide/encode how hands-off sessions populate `sessions.mode` (extend enum vs reuse `batch`).

- **M3 — `schema_runs.id TEXT` / `sessions.schema_run_id TEXT` deviates from the locked `uuid` key convention.** LLD-data-layer §("Keys") pins `id uuid primary key default gen_random_uuid()` and "natural keys carried as columns, not PKs"; every table in `001_init.sql` uses `uuid` PKs, and `sessions.id` is `uuid` (`001_init.sql:100`). The new `schema_runs.id TEXT PRIMARY KEY` + `sessions.schema_run_id TEXT … REFERENCES schema_runs(id)` (`docs/design/LLD-amend-005.md:66,78`) are **internally consistent** (TEXT→TEXT FK is valid — not a broken data model), but they break the established convention without justification. The §2/§6 entrypoint also implies app-generated ids (`-- uuid` comment on the TEXT column, line 66, and `store.create_schema_run` returning an id) — if the value is a uuid anyway, a `uuid` column is the consistent choice. Not blocking (the FK and index are coherent), but a quality/consistency gap a downstream consumer should know about.
  - Suggested fix unit: align `schema_runs.id` + `sessions.schema_run_id` to `uuid` (or record an explicit decision to use TEXT and why).

### LOW severity

- **L1 — Q7–Q10 are cited as numbered questions but do not exist in `QUESTIONS.md`.** The doc (and the ledger row, and D51) reference "Q7 … Q10" as the ratification answers, but `.gotm/QUESTIONS.md` only contains Q1–Q5; the 10-question R3 ratification answers live **inside D51's decision text**, not as discrete Q-entries. The cross-references still resolve *semantically* (D51 carries the Q7=never-auto-apply, Q8=persist-awaiting_input, Q9=schema_runs, Q10=app-triggered-Job content the doc relies on), so this is a citation-hygiene nit, not a fidelity break. Flagging because a reader following "Q7" to `QUESTIONS.md` finds nothing.

- **L2 — cross-doc migration ordering (003 here vs 002 in sibling LLD-amend-006), no collision.** This doc consistently uses `003` (§4, §10). The sibling LLD-amend-006 (also a pending R3 design unit) uses `002` for `comment_library` changes. The numbers are sequential and touch disjoint tables (`schema_runs`/`sessions` vs `comment_library`), so there is **no collision** — but the numbering implies amend-006's `002` lands before amend-005's `003` though amend-005 is the lower-numbered amendment doc. Build phase must apply `002` then `003`; harmless given disjoint tables, noted so the ordering is deliberate.

- **L3 — D48 identity extraction now also occurs at `POST /api/schema-runs`.** §2/§4/§5 set `schema_run.created_by` from `X-Forwarded-Email` (OBO identity) at run-creation. D48's 2026-06-12 refinement says `RequestIdentity` is extracted "**only** at the `/apply` route" and that session `created_by` = the SP. The doc's usage is a benign extension (it records who *initiated* the run on a **new** table column for attribution, never a UC write, and explicitly preserves SP-as-actor for generation per §9 R2) — it does not contradict the no-write boundary. But it does widen identity extraction beyond `/apply`; worth an explicit nod that this is an intended, attribution-only extension of D48.

### UNVERIFIED

- **U-1 — `geniefy_setup` vs `geniefy_schema_run` Job packaging (U59).** The doc says the new Job is "packaged like `geniefy_setup`" and cites U59 for packaging. The ledger row for U101 names U59 as an input, but U59's actual content (the `geniefy_setup` job definition in `databricks.yml`) was not provided in the oracle set; I confirmed `geniefy_setup` exists as a DAB job (U62 ledger row) but did not verify that a parameterized second Job follows the same packaging shape cleanly. Low-risk (it's a build-time concern), flagged for the build unit.
